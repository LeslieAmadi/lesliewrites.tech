BPMN Workflow Documentation

When a process crosses teams and systems, prose isn't enough you need a notation everyone reads the same way. This case study documents a customer-onboarding workflow in BPMN: pools and lanes for who does what, events for what starts and ends it, tasks for the work, and gateways for the decisions plus a plain-English guide so non-analysts can read it too.

The business problem

A process that spans three teams gets described three different ways each team draws its own part, the boxes mean different things on each diagram, and the handoffs between them fall into the gaps. When an analyst says “gateway” and an engineer pictures something different from what operations pictures, the diagram has failed at its one job: a shared, unambiguous understanding.

BPMN is a shared language, not decoration

The value of BPMN is that every shape has one agreed meaning. A diamond is always a decision; a thin circle always starts the process and a thick one always ends it; a lane always means “who is responsible.” Once a team adopts the notation, a diagram stops being one person's sketch and becomes a contract everyone reads identically. This case study documents a real workflow that way and teaches the notation alongside it.

The BPMN elements, in plain English

BPMN has dozens of symbols, but a clear, useful model needs only a handful. The documentation deliberately uses this core set enough to model almost any business process, few enough that a stakeholder can learn them in a minute.

ElementSymbolMeans
Start eventThin circleWhat kicks the process off
TaskRounded rectangleA unit of work someone does
Exclusive gatewayDiamond with ×A decision one path out
Parallel gatewayDiamond with +A split all paths at once
End eventThick circleWhere the process finishes
LaneHorizontal bandWho is responsible for the steps in it

The worked model customer onboarding

Here is a real onboarding process modelled in BPMN across three lanes Customer, Support, and System. Read it left to right; each lane shows who owns each step, and the arrows crossing lanes are the handoffs.

A BPMN model of customer onboarding. The thin circle starts it (Customer signs up); rounded rectangles are tasks in each responsible lane; the diamond is an exclusive gateway (“Verified?”) routing to either Support review or straight to provisioning; the thick circle ends it. Every arrow crossing a lane boundary is a handoff.

🔀

Gateways the decisions and splits

Gateways are where most process complexity (and most modelling mistakes) live. The documentation uses just two, and is strict about the difference.

GatewayBehaviourUse when
Exclusive (×)Exactly one path is takenA yes/no or either/or decision “Verified?”
Parallel (+)All paths run at onceIndependent work that can happen simultaneously e.g. “send welcome email” and “notify account manager”
⚠ Don't confuse a decision with a split

The most common BPMN error is using one gateway where the other belongs. An exclusive gateway chooses a path; a parallel gateway does all of them. Drawing a parallel split as an exclusive decision (or vice-versa) changes what the process actually does so the documentation calls this out explicitly as the thing to get right.

👤

Lanes & the handoffs between them

Lanes answer “who is responsible?” for every step, and the arrows that cross between lanes are the handoffs the single most failure-prone part of any cross-team process. Modelling in lanes forces every handoff to be drawn, not assumed.

LaneOwnsHands off to
CustomerSign up, submit detailsSystem (to create the account)
SupportReview & resolve failed verificationSystem (to provision once resolved)
SystemCreate account, provision, notifyCustomer (welcome email)
Every lane-crossing arrow is a risk worth naming

When work passes from one lane to another, it can stall, get dropped, or arrive incomplete. By making each handoff a visible arrow across a lane boundary, the model turns invisible coordination risk into something a team can see and design around the same instinct behind the rework-loop analysis in the process-improvement case study.

Modelling conventions

A notation only stays a shared language if everyone draws it the same way. These house rules keep every diagram consistent and readable.

Flow directionLeft to right, top to bottom always.
One start, clear endsA single start event; every path reaches an explicit end.
Label tasks as verbs“Create account,” not “Account” a task is an action.
Label gateway pathsEvery path out of a decision is labelled (yes/no, etc.).
Core symbols onlyStick to the handful everyone knows; avoid exotic shapes.

Key learnings

Use the smallest notation that works.

A handful of symbols models almost any process and stays readable; reaching for exotic BPMN elements loses the audience.

Lanes make handoffs honest.

Modelling by responsibility forces every cross-team handoff onto the page, where it can be managed.

Teach the notation alongside the model.

A diagram only creates shared understanding if everyone can read it so the key travels with the model.

📝 Writer's note

BPMN is powerful precisely because it's constrained: a fixed vocabulary of shapes that mean the same thing to everyone. The temptation is to show off the full notation; the discipline is to use the smallest subset that captures the process and to teach it as you go, so the diagram includes its own audience rather than excluding it.

What makes BPMN matter for the kind of work I do is the lane structure. The hardest part of a cross-team process is never a single step it's the handoffs, the moments where responsibility changes hands. Lanes force every one of those into a visible arrow, which is exactly where a process analyst earns their keep: turning invisible coordination risk into something a team can actually see and fix.
