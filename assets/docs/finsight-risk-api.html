The business problem

On a Tuesday afternoon in March, a fraud analyst at a mid-sized European payment gateway noticed something unusual: a cluster of transactions from a single IP range, each just below the manual review threshold, each individually passing velocity checks. By the time the pattern was flagged manually, forty-two transactions had cleared. Total exposure: €83,000.

This is the problem FinSight was built to solve not the obvious fraud, but the patient, rules-aware fraud that understands your thresholds and works just beneath them. FinSight evaluates every transaction against a machine learning risk model, returning a scored assessment in under 250 milliseconds fast enough to sit in the critical path of payment authorisation without adding customer-visible latency.

The documentation goal

What I set out to achieve

Create a documentation experience that enables engineers to understand the system architecture, perform a successful integration, troubleshoot issues independently, and deploy confidently into production without needing to open a support ticket or schedule a call with the engineering team. The secondary goal: give risk analysts and product owners enough clarity about the risk model that they can configure thresholds and interpret outputs with confidence.

System architecture

FinSight is a synchronous, stateless risk scoring service sitting between your payment gateway and the card network called after the customer confirms checkout, before the authorisation request is forwarded to the issuer. Its job is to produce a risk assessment fast enough that the customer never notices it happened.

1
Customer confirms checkout

Merchant sends authorisation request to your payment gateway

2
Gateway calls FinSight POST /v1/risk/score

Before forwarding to the card network synchronous call with 3s client timeout

3
FinSight Risk Engine evaluates

Feature extraction from Feature Store → ML model scoring → business rule application

4
Risk assessment returned P95 under 250ms

risk_score · risk_band · recommended_action · reasons · trace_id

5
Gateway applies your policy

allow → forward to issuer · challenge → enforce 3DS · block → decline immediately

Risk model

The risk score is not a rule engine. FinSight uses a machine learning model trained on historical transaction outcomes to assign a fraud probability. Rules are layered on top for velocity limits and block lists, but the score itself reflects learned statistical patterns which means it can detect fraud that no single rule would catch.

Signal categories

Signal categoryExamplesBusiness value
Transaction signalsAmount, currency, merchant category code, channel, time of dayEstablish baseline risk profile for this type of transaction at this merchant
Behavioural signalsVelocity on card (1h/6h/24h), account age days, deviation from customer patternsDetect sudden changes in usage that often precede fraud new-account fraud, account takeover
Device & network signalsDevice fingerprint, IP geolocation, proxy/VPN/Tor detection, billing country mismatchIdentify suspicious origin patterns the same card looks very different from a VPN exit node
Historical outcome signalsChargeback history on token, confirmed fraud in past 90 days, merchant-level fraud ratesThe model learns from what actually happened, not what rules predict should happen

Recommended action thresholds

allow
Score 0.00 – 0.30
Forward to card network for normal authorisation. No additional friction added to the customer journey.
challenge
Score 0.30 – 0.70
Enforce 3-D Secure step-up authentication before forwarding. Confirms identity without blocking legitimate transactions.
block
Score 0.70 – 1.00
Decline immediately and log a potential fraud event for analyst review. Thresholds are configurable per merchant via dashboard.
⚠ Probabilistic outputs important for risk teams

The risk score is a probability estimate, not a decision. A score of 0.82 means the model assigns an 82% probability that this transaction matches historically fraudulent patterns it does not mean the transaction is definitely fraudulent. A legitimate high-value purchase from a new device will often score elevated. That is correct model behaviour. Your policy layer determines what the score means in your specific customer context.

Quickstart — first successful score in 5 minutes

You need a FinSight API key and curl. Nothing else.

Step 1 — verify your credentials

curl · GET /v1/ping 
GET https://api.finsight.ai/v1/ping \
  -H "Authorization: Bearer sk_live_51Fx8C0Xexample123"

// Expected response — 200 OK
{
  "status":        "ok",
  "authenticated": true,
  "environment":   "production",
  "key_scope":     "full_access"
}
Seeing a 401?

Check you copied the full key including the sk_live_ prefix for production or sk_test_ for sandbox. Test keys cannot call production endpoints and return 401 regardless of format. Check for trailing spaces in your environment variable.

Step 2 — score a realistic transaction

This is a €289.50 e-commerce purchase structurally accurate to what a real integration sends. Note: amount is in cents — 28950 = €289.50. This is the most common integration mistake.

curl · POST /v1/risk/score 
POST https://api.finsight.ai/v1/risk/score \
  -H "Authorization: Bearer sk_live_51Fx8C0Xexample123" \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: idem_20250614_checkout_00491" \
  -d '{
    "transaction_id": "txn_493930",
    "occurred_at":   "2025-06-14T14:23:11Z",
    "amount":         28950,   // €289.50 in cents
    "currency":      "EUR",
    "merchant": { "id": "mrc_9102", "category": "ecommerce_electronics", "country": "DE" },
    "payment_method": { "type": "card", "bin": "424242", "last4": "0018", "token_id": "card_tok_91f8a" },
    "customer": { "id": "cust_1203", "account_age_days": 340 },
    "device": { "ip_address": "81.2.69.160", "device_id": "dev_338192" }
  }'
Response · 200 OK · Low risk — established account, known device 
{
  "transaction_id":      "txn_493930",
  "risk_score":          0.23,
  "risk_band":           "low",
  "recommended_action": "allow",
  "reasons": [ "established_account", "known_device", "ip_country_matches_billing" ],
  "model_version":  "finsight-risk-3.1",
  "trace_id":       "req_f91ec7d02a",
  "processed_at":  "2025-06-14T14:23:11.347Z"
}
Try this watch the model respond in real time

Change account_age_days to 2, set device_id to any new value, and raise the amount to 189900. Score it again. Watch the risk_score climb and the recommended_action shift to challenge or block. That is the model correctly responding to a new account + unknown device + high-value combination one of the strongest signals for new account fraud.

Error handling

Every error response includes a machine-readable error code, human-readable message, the affected parameter, and a trace ID. Log the trace ID with every error it is the fastest path to diagnosis.

Error response shape — all error codes follow this structure 
{
  "error": {
    "code":    "4001_INVALID_SCHEMA",
    "message": "Field 'amount' must be a positive integer in the smallest currency unit.",
    "param":   "amount",
    "trace_id":"req_abc123",
    "doc_url": "https://docs.finsight.ai/errors/4001"
  }
}
HTTPError codeCommon causeRecommended action
400 4001_INVALID_SCHEMA Missing or malformed field. Check the param field to identify which field failed. Fix the request payload before retrying. Retrying without fixing produces the same error.
401 4010_UNAUTHENTICATED Missing or invalid API key. Using a test key on production endpoint. Check Authorization: Bearer sk_live_… format. Verify environment. Rotate immediately if compromised.
429 4290_RATE_LIMIT Request rate exceeded for this API key. Respect the Retry-After header. Implement exponential backoff with jitter never fixed-interval retry.
500 5000_SERVER_ERROR Unexpected error on FinSight's side. Rare, typically transient. Retry once after 1 second with the same idempotency key. After three retries, fail open and log the trace_id.
⚡ Fail-open policy agree this before go-live

FinSight sits in the critical path of payment authorisation. Blocking all transactions during a FinSight outage stops your revenue along with the fraud. The recommended approach: fail open on persistent 5xx errors, allow the transaction, log the unavailability event, and review those transactions in your fraud queue after the outage resolves. This is a business decision document it explicitly in your runbook before launching.

Idempotency

Real scenario why this matters in production
1
Customer clicks "Pay now." Your gateway sends a risk score request to FinSight.
2
The network drops before the response arrives. Your gateway times out at 3 seconds.
3
Without idempotency: retry creates a second evaluation. Real-time signals may have shifted you could receive a different score and a different recommended action for the same transaction.
4
With idempotency: retrying with the same key returns the original response. No second evaluation, no ambiguity, no inconsistent action on a real-money transaction.

Include an idempotency key on every call to POST /v1/risk/score. FinSight stores keys for 72 hours any request with the same key within that window returns the original response without re-scoring.

Request header 
Idempotency-Key: idem_20250614_checkout_00491

Production integration checklist

Every item here represents something that has caused a production incident on at least one integration. Work through this before routing live customer transactions through FinSight.

  • ✓
    Amounts are in the smallest currency unit 28950 for €289.50, not 289.50. No error is thrown for wrong units — signals will simply be incorrect.
  • ✓
    API key is in a secrets manager, not hardcoded in application config or committed to version control.
  • ✓
    Idempotency keys implemented for all risk score requests and stored for the duration of your retry window.
  • ✓
    3-second client timeout with one retry on 5xx. FinSight's P95 is under 250ms, 3 seconds provides generous headroom without risking customer-visible delays.
  • ✓
    Error handling covers all classes 4xx are not retried (fix the request), 429 respects Retry-After, 5xx fails open after three retries.
  • ✓
    trace_id logged alongside your internal transaction ID in every response success and error.
  • ✓
    Device signals included ip_address and device_id. Scoring without device signals produces materially weaker new-account fraud detection.
  • ✓
    Chargeback feedback endpoint integrated will fire within 48 hours of a confirmed chargeback. This is how the model improves over time for your context.
  • ✓
    Risk band distribution baselined in staging before production go-live. You need a reference to detect unexpected changes after launch.
  • ✓
    Fail-open policy agreed with your risk team and documented in your runbook. Ambiguity about what to do during outages produces inconsistent production behaviour.
  • ✓
    Staging tested with realistic edge cases new account + new device + high value; known card + unusual country; legitimate transaction from known fraud IP range.

Key learnings

Documentation must reflect user goals, not system structure

Developers rarely read documentation sequentially. They seek immediate answers to specific questions. Structuring content around user goals (authenticate, make first call, handle errors, go live) rather than system components (API gateway, risk engine, feature store) significantly improves usability and reduces the time to successful integration.

Examples drive adoption faster than explanations

Clear, realistic examples with working curl commands including the exact wrong things people do (wrong amount unit, wrong environment key, missing device signals) prevented more support tickets than any amount of conceptual explanation. The best documentation anticipates the failure before it happens.

Documentation is part of the product not a post-release activity

Developer documentation directly influences product adoption, support volume, implementation success, and overall developer experience. Documentation written in parallel with engineering not after it is more accurate, more complete, and gets used more. When I joined the FinSight sprint cycle at the design stage rather than the release stage, the quality of every document improved and the review cycles shortened.
