The business problem

On a Tuesday afternoon in March, a fraud analyst at a mid-sized European payment gateway noticed something unusual: a cluster of transactions from a single IP range, each just below the manual review threshold, each individually passing velocity checks. By the time the pattern was flagged manually, forty-two transactions had cleared. Total exposure: €83,000.

This is the problem FinSight was built to solve — not the obvious fraud, but the patient, rules-aware fraud that understands your thresholds and works just beneath them. FinSight evaluates every transaction against a machine learning risk model, returning a scored assessment in under 250 milliseconds — fast enough to sit in the critical path of payment authorisation without adding customer-visible latency.

The documentation goal

What I set out to achieve

Create a documentation experience that enables engineers to understand the system architecture, perform a successful integration, troubleshoot issues independently, and deploy confidently into production — without needing to open a support ticket or schedule a call with the engineering team. The secondary goal: give risk analysts and product owners enough clarity about the risk model that they can configure thresholds and interpret outputs with confidence.

Risk model

The risk score is not a rule engine. FinSight uses a machine learning model trained on historical transaction outcomes to assign a fraud probability. Rules are layered on top for velocity limits and block lists, but the score itself reflects learned statistical patterns — which means it can detect fraud that no single rule would catch.

Signal categories

Signal categoryExamplesBusiness value
Transaction signalsAmount, currency, merchant category code, channel, time of dayEstablish baseline risk profile for this type of transaction at this merchant
Behavioural signalsVelocity on card (1h/6h/24h), account age days, deviation from customer patternsDetect sudden changes in usage that often precede fraud — new-account fraud, account takeover
Device & network signalsDevice fingerprint, IP geolocation, proxy/VPN/Tor detection, billing country mismatchIdentify suspicious origin patterns — the same card looks very different from a VPN exit node
Historical outcome signalsChargeback history on token, confirmed fraud in past 90 days, merchant-level fraud ratesThe model learns from what actually happened, not what rules predict should happen

Recommended action thresholds

allow
Score 0.00 – 0.30
Forward to card network for normal authorisation. No additional friction added to the customer journey.
challenge
Score 0.30 – 0.70
Enforce 3-D Secure step-up authentication before forwarding. Confirms identity without blocking legitimate transactions.
block
Score 0.70 – 1.00
Decline immediately and log a potential fraud event for analyst review. Thresholds are configurable per merchant via dashboard.
⚠ Probabilistic outputs — important for risk teams

The risk score is a probability estimate, not a decision. A score of 0.82 means the model assigns an 82% probability that this transaction matches historically fraudulent patterns — it does not mean the transaction is definitely fraudulent. A legitimate high-value purchase from a new device will often score elevated. That is correct model behaviour. Your policy layer determines what the score means in your specific customer context.

Error handling

Every error response includes a machine-readable error code, human-readable message, the affected parameter, and a trace ID. Log the trace ID with every error — it is the fastest path to diagnosis.

Error response shape — all error codes follow this structure 
{
  "error": {
    "code":    "4001_INVALID_SCHEMA",
    "message": "Field 'amount' must be a positive integer in the smallest currency unit.",
    "param":   "amount",
    "trace_id":"req_abc123",
    "doc_url": "https://docs.finsight.ai/errors/4001"
  }
}
HTTPError codeCommon causeRecommended action
400 4001_INVALID_SCHEMA Missing or malformed field. Check the param field to identify which field failed. Fix the request payload before retrying. Retrying without fixing produces the same error.
401 4010_UNAUTHENTICATED Missing or invalid API key. Using a test key on production endpoint. Check Authorization: Bearer sk_live_… format. Verify environment. Rotate immediately if compromised.
429 4290_RATE_LIMIT Request rate exceeded for this API key. Respect the Retry-After header. Implement exponential backoff with jitter — never fixed-interval retry.
500 5000_SERVER_ERROR Unexpected error on FinSight's side. Rare, typically transient. Retry once after 1 second with the same idempotency key. After three retries, fail open and log the trace_id.
⚡ Fail-open policy — agree this before go-live

FinSight sits in the critical path of payment authorisation. Blocking all transactions during a FinSight outage stops your revenue along with the fraud. The recommended approach: fail open on persistent 5xx errors, allow the transaction, log the unavailability event, and review those transactions in your fraud queue after the outage resolves. This is a business decision — document it explicitly in your runbook before launching.

Idempotency

Real scenario — why this matters in production
1
Customer clicks "Pay now." Your gateway sends a risk score request to FinSight.
2
The network drops before the response arrives. Your gateway times out at 3 seconds.
3
Without idempotency: retry creates a second evaluation. Real-time signals may have shifted — you could receive a different score and a different recommended action for the same transaction.
4
With idempotency: retrying with the same key returns the original response. No second evaluation, no ambiguity, no inconsistent action on a real-money transaction.

Include an idempotency key on every call to POST /v1/risk/score. FinSight stores keys for 72 hours — any request with the same key within that window returns the original response without re-scoring.

Request header 
Idempotency-Key: idem_20250614_checkout_00491

Production integration checklist

Every item here represents something that has caused a production incident on at least one integration. Work through this before routing live customer transactions through FinSight.

  • ✓
    Amounts are in the smallest currency unit — 28950 for €289.50, not 289.50. No error is thrown for wrong units — signals will simply be incorrect.
  • ✓
    API key is in a secrets manager, not hardcoded in application config or committed to version control.
  • ✓
    Idempotency keys implemented for all risk score requests and stored for the duration of your retry window.
  • ✓
    3-second client timeout with one retry on 5xx. FinSight's P95 is under 250ms — 3 seconds provides generous headroom without risking customer-visible delays.
  • ✓
    Error handling covers all classes — 4xx are not retried (fix the request), 429 respects Retry-After, 5xx fails open after three retries.
  • ✓
    trace_id logged alongside your internal transaction ID in every response — success and error.
  • ✓
    Device signals included — ip_address and device_id. Scoring without device signals produces materially weaker new-account fraud detection.
  • ✓
    Chargeback feedback endpoint integrated — will fire within 48 hours of a confirmed chargeback. This is how the model improves over time for your context.
  • ✓
    Risk band distribution baselined in staging before production go-live. You need a reference to detect unexpected changes after launch.
  • ✓
    Fail-open policy agreed with your risk team and documented in your runbook. Ambiguity about what to do during outages produces inconsistent production behaviour.
  • ✓
    Staging tested with realistic edge cases — new account + new device + high value; known card + unusual country; legitimate transaction from known fraud IP range.

Key learnings

Documentation must reflect user goals, not system structure

Developers rarely read documentation sequentially. They seek immediate answers to specific questions. Structuring content around user goals (authenticate, make first call, handle errors, go live) rather than system components (API gateway, risk engine, feature store) significantly improves usability and reduces the time to successful integration.

Examples drive adoption faster than explanations

Clear, realistic examples with working curl commands — including the exact wrong things people do (wrong amount unit, wrong environment key, missing device signals) — prevented more support tickets than any amount of conceptual explanation. The best documentation anticipates the failure before it happens.

Documentation is part of the product — not a post-release activity

Developer documentation directly influences product adoption, support volume, implementation success, and overall developer experience. Documentation written in parallel with engineering — not after it — is more accurate, more complete, and gets used more. When I joined the FinSight sprint cycle at the design stage rather than the release stage, the quality of every document improved and the review cycles shortened.

The internal architecture separates the API gateway (authentication, rate limiting, schema validation), the risk engine (feature extraction, model inference, business rule evaluation), and the feature store (historical behavioural data per card token, device, merchant, and IP). These layers scale independently. The API gateway is horizontally scaled behind a load balancer. The risk engine uses dedicated inference infrastructure. The feature store is a low-latency key-value store with millisecond read times.

What this means for your integration: FinSight behaves identically under ten requests per minute and ten thousand. You do not need to implement back-pressure logic for normal traffic patterns. Rate limiting is applied at the API key level and documented in the monitoring section below.

Risk Model — What FinSight Is Actually Looking At

The risk score is not a rule engine. FinSight uses a machine learning model trained on historical transaction outcomes — including confirmed fraud, chargebacks, and successful legitimate transactions — to assign a probability that a given transaction represents fraud. Rules are layered on top of the model to handle velocity limits and explicit block lists, but the score itself reflects learned statistical patterns, not a checklist.

This matters because it means the system can detect patterns that no single rule would catch. A transaction might individually look legitimate across every individual check — reasonable amount, known device, matching billing address — but still score high risk because the combination of signals matches patterns associated with fraud at the population level. Conversely, a high-value transaction from an unfamiliar device might score low risk because the customer's account history and behavioural profile are strong.

Signal categories

  • Transaction signals Amount, currency, merchant category code, channel (web / mobile / in-person), time of day, geographic location of merchant. These establish the baseline risk profile for this type of transaction at this merchant.
  • Behavioural signals Transaction velocity on this card token over the past 1, 6, 24, and 72 hours. Time since the card was first seen. Deviation from the customer's typical transaction patterns. These catch sudden changes in usage that often precede fraud.
  • Device & network signals Device fingerprint, IP address, IP geolocation, proxy/VPN/Tor detection, distance between IP geolocation and billing address. A customer making a large purchase from a VPN exit node in a different country than their registered address is a different risk profile than one making the same purchase from their home broadband.
  • Historical outcome signals Chargeback history associated with this card token, customer ID, device, and merchant. Confirmed fraud events in the preceding 90 days. These signals are the model learning from what actually happened, not what rules predict.
Important: probabilistic outputs

The risk score is a probability estimate, not a decision. A score of 0.82 means the model assigns an 82% probability that this transaction matches patterns historically associated with fraud — it does not mean the transaction is definitely fraudulent. The recommended_action field translates that probability into an actionable category based on configurable thresholds, but your policy layer determines what those actions mean in your context. A legitimate high-value transaction from a new device will often score elevated. That is correct model behaviour. Your policy should account for this.

API Reference

POST /v1/risk/score

The primary endpoint. Called synchronously during checkout. Must complete within your gateway's timeout budget — we recommend a 3-second client-side timeout with one retry on 5xx responses.

Request fields

  • transaction_id requiredYour unique identifier for this transaction. Used to correlate FinSight scoring with your internal records and to support idempotency. String, max 64 characters.
  • occurred_at requiredTransaction timestamp in ISO 8601 UTC format. FinSight uses this to calculate time-based velocity signals correctly. Do not send the server time — send the time the customer initiated the transaction.
  • amount requiredTransaction amount in the smallest currency unit — cents for EUR/GBP/USD, pence for GBP. Integer. Sending 28950 represents €289.50. Sending 289 would represent €2.89 and will produce incorrect risk signals.
  • currency requiredISO 4217 three-letter currency code. String.
  • merchant.id requiredYour internal merchant identifier. Used to build merchant-level behavioural baselines.
  • merchant.category requiredMerchant category string. Used for category-level risk calibration. See the merchant category list in the supplementary reference.
  • merchant.country requiredISO 3166-1 alpha-2 country code for the merchant's registered location.
  • payment_method.token_id requiredYour tokenised card identifier — never send raw card numbers. FinSight uses this to build card-level velocity and history signals.
  • payment_method.bin optionalFirst six digits of the card number. Improves issuer-level risk signals. Strongly recommended.
  • customer.id optionalYour internal customer identifier. Used to build customer-level behavioural baselines across card tokens.
  • customer.account_age_days optionalDays since the customer created their account with you. One of the most predictive signals for new account fraud. Always include this if you have it.
  • device.ip_address optionalCustomer's IP address. Used for geolocation, proxy/VPN detection, and device fingerprinting. Do not send internal load balancer IPs — send the customer-facing IP from the X-Forwarded-For header.
  • device.device_id optionalYour device fingerprint identifier. The more consistent and accurate this is across sessions, the more useful FinSight's device-level signals become.

Response fields

  • risk_scoreFloat between 0 and 1. Higher values indicate higher estimated fraud probability. 0.0–0.30 is typically low risk. 0.30–0.70 is elevated. Above 0.70 is high risk. These thresholds are configurable via your FinSight dashboard.
  • risk_bandDiscrete category corresponding to your configured thresholds. One of: low, medium, high. Easier to use in policy logic than the raw score.
  • recommended_actionFinSight's suggested response based on your configured policy. One of: allow, challenge, block. This is a recommendation — your policy layer makes the final decision.
  • reasonsArray of string codes explaining the top signals that drove the score. Designed for analyst tooling and audit logging. See the reasons reference for the full list of possible values.
  • model_versionIdentifier of the model version that scored this transaction. Include in your logs — it is essential for debugging when model behaviour changes after an update.
  • trace_idUnique request identifier. Include in all support queries. Log it alongside your internal transaction identifier so you can correlate FinSight logs with your own.

POST /v1/events/chargeback

Send confirmed chargebacks back to FinSight so the model can learn from outcomes in your specific transaction context. This is how FinSight becomes more accurate for your business over time. Send this within 48 hours of a chargeback being confirmed — not at dispute initiation.

Error Handling

FinSight errors follow a consistent structure. Every error response includes the HTTP status code, a machine-readable error code, a human-readable message, and a trace ID. Log the trace ID with every error — it is the fastest path to diagnosis. 

{
  "error": {
    "code": "4001_INVALID_SCHEMA",
    "message": "Field 'amount' must be a positive integer representing the amount in the smallest currency unit.",
    "param": "amount",
    "trace_id": "req_abc123",
    "doc_url": "https://docs.finsight.ai/errors/4001"
  }
}
400 — Schema errors

The request body does not match the expected schema. Check the param field to identify which field failed. Fix the request before retrying — retrying a 400 without fixing the payload will produce the same error indefinitely.

401 — Authentication failure

Missing or invalid API key. Check your Authorization header format (Bearer sk_live_...). Verify you are using the correct key for the correct environment — test keys cannot call production endpoints.

429 — Rate limit exceeded

You have exceeded the request rate for your API key. The response includes a Retry-After header specifying how many seconds to wait. Implement exponential backoff with jitter rather than a fixed retry interval.

500 — Server errors

An unexpected error occurred on FinSight's side. These are rare and typically transient. Retry once after 1 second with the same idempotency key. If the error persists across three retries, fail open (allow the transaction) and log the trace ID for support.

Fail-open policy for 5xx errors

FinSight sits in the critical path of payment authorisation. If FinSight is temporarily unavailable and you block all transactions as a result, you stop your revenue along with the fraud. The recommended approach is to fail open on persistent 5xx errors — allow the transaction, log the FinSight unavailability event, and review those transactions in your fraud queue after the outage resolves. This is a business decision and should be agreed with your risk team before going live.

Idempotency — Why It Matters and How to Use It

Here is a scenario that happens more often than you might expect: a customer clicks "Pay now." Your gateway sends a risk score request to FinSight. The network drops before the response arrives. Your gateway times out. Should you retry?

Without idempotency, retrying creates a second risk scoring event for the same transaction. If the model processes it differently (which can happen with real-time signals), you may get a different score — and a different action — for the same transaction. With idempotency, retrying the same request with the same key returns the original response, no second evaluation, no ambiguity.

Every call to POST /v1/risk/score should include an idempotency key: 

Idempotency-Key: idem_20250614_checkout_00491

Use your internal transaction ID as the idempotency key, or a deterministic combination of customer ID and timestamp. Keep idempotency keys unique per transaction and store them for the duration of your retry window (typically 24 hours). FinSight stores idempotency keys for 72 hours — any request with the same key within that window returns the original response without re-scoring.

What to instrument on your side

FinSight cannot tell you when your integration is behaving unexpectedly — only when its own service is degraded. The following metrics on your gateway will catch integration issues that FinSight's own monitoring will not surface:

  • FinSight call latency (P95, P99)Baseline this in staging before going live. Any sustained increase in production is worth investigating — it may indicate network issues between your gateway and FinSight's edge.
  • 4xx rate by error codeA sudden spike in 400 errors often means a schema change on your side — a field that was previously optional becoming required, or a type mismatch in your payload builder.
  • Risk band distributionIf the proportion of high risk transactions suddenly increases or collapses, something has changed — either in your traffic (new fraud pattern, new customer segment) or in how you are sending data (missing device signals, incorrect amounts).
  • recommended_action distributionTrack the proportion of allow / challenge / block over time. If challenge volume increases unexpectedly, it will be visible in your 3DS conversion rates before it shows up anywhere else.

Security

  • TransportAll requests must use TLS 1.2 or higher. HTTP requests are rejected at the gateway with a redirect to HTTPS — do not depend on the redirect in production. Configure your HTTP client to enforce HTTPS.
  • API key storageStore API keys in a secrets manager (AWS Secrets Manager, GCP Secret Manager, HashiCorp Vault) or environment variables. Never commit keys to version control, log files, or error messages. Rotate immediately on any suspected exposure.
  • PII minimisationSend internal token IDs and identifiers rather than raw card numbers, customer names, or email addresses. FinSight does not require PII to generate accurate risk scores. Sending less data reduces your regulatory surface area.
  • Audit trailsLog the trace_id and model_version from every FinSight response alongside your internal transaction record. These are the primary inputs to any post-hoc fraud investigation or regulatory audit.

Integration Checklist

Work through this checklist before routing live customer transactions through FinSight. Every item here represents something that has caused a production incident on at least one integration.

  • Amounts are in the smallest currency unit — 28950 for €289.50, not 289.50. The single most common integration mistake.
  • API key is stored in a secrets manager, not hardcoded in application config or committed to version control.
  • Idempotency keys are implemented for all risk score requests and stored for the duration of your retry window.
  • Client-side timeout is set to 3 seconds with one retry on 5xx. FinSight's P95 latency is under 250ms — a 3-second timeout provides generous headroom without risking customer-visible delays.
  • Error handling covers all error classes — 4xx errors are not retried (fix the request), 429 errors respect the Retry-After header, 5xx errors fail open after three retries.
  • The trace_id is logged alongside your internal transaction identifier in every response, success or error.
  • Device signals are included — ip_address and device_id. Risk scoring without device signals produces materially weaker results for new-account fraud.
  • The chargeback feedback endpoint is integrated and will be called within 48 hours of a confirmed chargeback.
  • Risk band distribution is baselined in staging before production go-live. You need a reference point to identify unexpected changes after launch.
  • Fail-open policy is agreed with your risk team and documented. Ambiguity about what to do when FinSight is unavailable will produce inconsistent behaviour in production.
  • The staging environment is tested with realistic transaction patterns — not just the happy path. Specifically: new account + new device + high value, known card + unusual country, legitimate-looking transaction from a known fraud IP range.
