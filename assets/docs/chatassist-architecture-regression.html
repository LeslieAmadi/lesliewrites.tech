ChatAssist Architecture

A customer-support assistant is only as trustworthy as the facts behind its answers. I documented the architecture of ChatAssist a multilingual (English–German) support assistant built on retrieval-augmented generation covering the request flow, how answers are grounded in a real knowledge base, intent recognition, escalation to humans, and the guardrails that keep it safe.

The business problem

A support team is drowning in repetitive questions in two languages, and the obvious fix “put an LLM on it” carries a serious risk: a raw language model will answer confidently even when it's wrong, inventing refund policies and product details that don't exist. In customer support, a confident wrong answer is worse than no answer, because the customer acts on it.

Why the architecture is the answer, not just the model

ChatAssist solves this with retrieval-augmented generation: instead of answering from the model's memory, every response is grounded in passages retrieved from the company's real knowledge base. The architecture not the model alone is what makes the assistant trustworthy. This document is the map of how every piece fits so the answer is always traceable to a source.

The request flow

A user message travels through a defined pipeline before any answer is generated. The orchestration layer is the conductor: it detects language and intent, retrieves grounding, calls the model, and applies guardrails before a word reaches the user.

The request flow. The orchestrator detects language and intent, the retriever pulls relevant passages from the knowledge base, the LLM answers using only that context, and guardrails verify and cite before replying. If nothing relevant is retrieved, ChatAssist escalates to a human instead of guessing.

🔍

Retrieval & grounding

Grounding is what separates ChatAssist from a chatbot that makes things up. The knowledge base is chunked into passages, each converted to a vector embedding and indexed; at query time the user's question is embedded the same way and the closest passages are retrieved and handed to the model as context. The model is instructed to answer only from that context.

Retrieval has two phases. Offline, the knowledge base is chunked, embedded, and indexed in a vector store. Live, the user's question is embedded the same way and matched against that index; the closest passages become the grounding context the model must answer from.

The grounding rule that prevents hallucination

The model is prompted to answer strictly from retrieved passages and to say it doesn't know when they don't contain the answer. That single architectural rule answer from context, escalate otherwise is what turns a creative language model into a reliable support agent.

Multilingual handling

ChatAssist serves English and German users from one knowledge base. The documentation makes the language strategy explicit so it stays consistent as the KB grows.

StageHow language is handled
DetectionThe orchestrator detects the user's language from the message.
RetrievalMultilingual embeddings let a question in one language retrieve passages in either.
GenerationThe model is instructed to answer in the user's language, grounded in the retrieved context.
EvaluationQuality is measured per language so neither degrades unnoticed (see the v2.0 evaluation report).

Escalation to humans

The most important thing a support bot can know is when to stop. ChatAssist escalates handing the conversation to a human agent with full context rather than guessing. The documentation defines the triggers explicitly.

TriggerWhy escalate
No relevant passages retrievedNothing to ground an answer in guessing would risk a wrong answer.
Low model / retrieval confidenceThe system isn't sure enough to be trusted with the reply.
Sensitive intent (billing dispute, complaint)A human should own emotionally or financially significant cases.
Explicit user requestThe user asks for a person always honoured.
Escalation is a feature, not a failure

A bot that escalates cleanly with full context earns more trust than one that tries to handle everything and gets some of it wrong. Documenting escalation as a designed path not an error state was central to making ChatAssist safe to put in front of customers.

Key learnings

The architecture makes it trustworthy, not the model.

RAG plus the answer-from-context rule is what turns a creative LLM into a reliable support agent. Documenting that rule was the most important page.

Knowing when to stop is the key feature.

Designed escalation with context earns more trust than a bot that attempts everything.

If you can't observe it, you can't trust it.

Retrieval hit rate and citation coverage turn “it seems fine” into something monitored.

📝 Writer's note

The hard part of documenting an AI system is resisting the magic. It's tempting to describe ChatAssist as “the model answers questions,” but that hides the architecture doing the real work retrieval, grounding, guardrails, escalation. I structured the whole reference around the journey of a single message so the reader sees exactly where trust is engineered in.

The grounding rule and the escalation triggers are the two sections I weighted most, because together they're the answer to the question every stakeholder really asks: “what stops it from making things up?” Documenting those as first-class architectural decisions is what lets a business put an AI assistant in front of real customers.
