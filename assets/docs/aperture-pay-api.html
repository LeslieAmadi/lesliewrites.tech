Aperture Pay Payments API

A REST API for charging, reversing, and reconciling payments against customer wallets with HMAC-signed requests, idempotent capture, and asynchronous settlement webhooks. This reference takes a developer from first signed call to edge-case recovery without leaving questions unanswered.

Overview

Aperture Pay sits between your checkout and a customer's funded wallet. Your server charges a wallet synchronously and receives an authoritative balance in the response. Final settlement clearing funds and applying any incentives happens asynchronously, and Aperture notifies your server through a signed webhook.

Two directions of traffic exist, and both are signed:

Requests you send

Creating charges, issuing reversals, reading state. You sign these with your secret.

Webhooks Aperture sends

Settlement and reversal outcomes. Aperture signs these; you verify them.

Charge lifecycle

Charge lifecycle
customer checkout
   │
   ▼
[1] POST /charges  ────►  Aperture authorizes against the wallet
   │                        returns 201 with an authoritative balance
   ▼
[2] you show the result to the customer (succeeded / declined)
   │
   ▼
[3] charge.settled  ◄────  Aperture webhook (asynchronous, signed)
   │                        funds cleared; cashback applied if any
   ▼
[4] reconcile in your ledger
ℹ A 201 is a commitment

A charge that returns 201 is committed against the wallet immediately. The settlement webhook confirms clearing and carries the final figures it does not re-authorize the charge.

Authenticating requests

Every request to Aperture carries three headers. The signature is an HMAC-SHA256 of a canonical string built from the request, keyed by your API secret and base64-encoded.

HeaderDescription
Aperture-KeyYour publishable key id, e.g. apk_live_8Qz…. Identifies the secret used to sign.
Aperture-TimestampUnix time in seconds at the moment of signing. Aperture rejects skew greater than ±300s.
Aperture-SignatureVersion-prefixed signature, e.g. v1=<base64>.

The canonical string

Concatenate four fields with a single newline (\n) between each, in this exact order. The body hash is the lowercase hex SHA-256 of the raw request body; for requests with no body, hash the empty string.

Canonical string
canonical = METHOD        # uppercase, e.g. POST
   + "\n" + PATH          # request path incl. query, e.g. /v1/charges
   + "\n" + BODY_SHA256   # hex sha256 of the raw body
   + "\n" + TIMESTAMP     # same value sent in Aperture-Timestamp

signature = base64( HMAC_SHA256( api_secret, canonical ) )
header    = "v1=" + signature
Worked example signing a charge
methodPOST
path/v1/charges
body sha2569f2c1b…a07e (sha256 of the JSON body)
timestamp1733930415
v1 signatureb3JpZ2luYWwtc2lnbmF0dXJl…K9xQ==
⚠ Sign the raw bytes you transmit

If you re-serialize the JSON after signing (changing key order or whitespace), the body hash won't match and Aperture returns 401 signature_mismatch.

Verifying webhooks Aperture sends you

Inbound webhooks use the same construction, signed with your secret, in the Aperture-Webhook-Signature header. Rebuild the canonical string from the received request and compare with a constant-time equality check. Reject anything outside the timestamp window.

Idempotency

Network failures make it impossible to know whether a charge committed. Retrying safely requires an idempotency key: a unique value you attach to a write so Aperture performs the operation at most once.

FieldTypeDescription
Idempotency-Keystring requiredA unique key per logical operation a UUID is ideal. Reusing a key replays the original outcome.
  • The first request with a key executes normally and the result is stored for 24 hours.
  • A replay with the same key returns the original status and body, plus Aperture-Idempotent-Replay: true even if the original failed.
  • Reusing a key with a different body returns 409 idempotency_conflict.
ℹ Only mutating verbs consume keys

POST requests consume idempotency keys. GET requests are already safe to retry and ignore the header.

Charges

Create a charge

Debits a customer wallet and returns the post-charge balance. Committed on a 201 response.

currencystring requiredISO 4217 code, lowercase, e.g. eur.

descriptionstring optionalHuman-readable note shown in the customer's statement.

Reversals

A reversal returns funds to the wallet and unwinds the charge. Use it for refunds, cancellations, or to recover from a failure on your side after a charge committed. Reversals are themselves idempotent and may be partial.

POST/v1/charges/{charge_id}/reversals
FieldTypeDescription
amountinteger optionalMinor units to return. Omit to reverse the full remaining amount. Must not exceed it.
reasonstring optionalOne of requested_by_customer, duplicate, fraud, operator_recovery.
Request
POST /v1/charges/chg_7H2nQ4xL/reversals HTTP/1.1
Host: api.aperturepay.dev

Aperture-Timestamp: 1733931000
Aperture-Signature: v1=cmV2ZXJzYWwtc2ln…Tt4=
Idempotency-Key: 9a77c0e2-a4d3-18b0-4f0b-2e641c5d9f7a

{ "amount": 2500, "reason": "requested_by_customer" }

⚠ Reversals claw back cashback

Reversing a charge whose cashback already settled also claws back the cashback. The net wallet movement is reported in balance_after and in the reversal.completed webhook so your ledger stays exact.

Settlement webhooks

Aperture delivers events to the endpoint registered on your account via signed POST. Your endpoint must be public HTTPS and must return a 2xx within 8 seconds. Anything else is treated as a delivery failure and retried.

EventWhen it fires
charge.settledFunds cleared. Carries final amounts and any applied cashback.
charge.failedA previously-succeeded charge could not clear (e.g. wallet closed). You should reverse downstream effects.
reversal.completedA reversal finished clearing.
cashback.grantedA campaign credited cashback to the wallet for one of your charges.

Event payload

POST /your/webhook/endpoint
POST /your/webhook/endpoint HTTP/1.1
Aperture-Webhook-Signature: v1=ZXZlbnQtc2lnbmF0dXJl…Qb8=
Aperture-Timestamp: 1733931120
Content-Type: application/json

{
  "id": "evt_5RcD8mK2",
  "type": "charge.settled",
  "created_at": "2025-12-11T14:12:00Z",
  "data": {
    "charge_id": "chg_7H2nQ4xL",
    "reference": "order_A1043",
    "amount": 2500,
    "currency": "eur",
    "cashback": { "campaign_id": "cmp_Wq1", "amount": 125 },
    "settled_at": "2025-12-11T14:12:00Z"
  }
}
ℹ Webhooks are at-least-once

Deduplicate on event.id and treat handlers as idempotent a settled charge may be delivered more than once.

Delivery & recovery

If your endpoint fails to acknowledge an event, Aperture retries on an exponential back-off. Each attempt re-signs with a fresh timestamp; the event.id is stable across attempts.

AttemptDelay after previousCumulative
1immediate0s
230s30s
32m~2m
410m~12m
51h~1h
66h~7h
7 (final)24h~31h

After the final attempt the event is marked undelivered. You can list and replay missed events at any time:

Recover undelivered events
GET  /v1/events?status=undelivered
POST /v1/events/{event_id}/redeliver
⚠ Don't gate checkout on the webhook

The synchronous 201 from POST /charges is the source of truth for whether to fulfil the order; webhooks reconcile the ledger afterward.

Cashback campaigns

A campaign returns a percentage of qualifying charges to the customer's wallet as cashback. Aperture evaluates campaigns automatically at settlement you don't compute incentives yourself.

POST/v1/campaigns
FieldTypeDescription
namestring requiredInternal label for the campaign. Max 64 chars.
rate_bpsinteger requiredCashback rate in basis points 500 = 5%.
currencystring requiredCharges in other currencies are ignored by this campaign.
max_cashbackinteger optionalPer-charge cap in minor units.
starts_atstring requiredISO-8601 start time.
ends_atstring requiredISO-8601 end time.
eligible_referencesarray optionalIf set, only charges whose reference has one of these prefixes qualify.

When a qualifying charge settles, the charge.settled webhook includes a cashback object and a separate cashback.granted event is emitted. Cashback is clawed back automatically if the underlying charge is later reversed.

Error reference

Errors return a non-2xx status and a structured body. The type groups the failure; the code is stable and safe to branch on; message is human-readable and may change.

Error shape
{
  "error": {
    "type": "invalid_request",
    "code": "missing_parameter",
    "message": "A required field is absent.",
    "param": "amount"
  }
}
StatusCodeMeaning
400missing_parameterA required field is absent.
400invalid_currencyUnsupported or malformed currency.
401signature_mismatchSignature did not match the canonical string.
401timestamp_skewTimestamp skew exceeded ±300s.
402insufficient_fundsWallet balance below charge amount.
404not_foundNo object with that id.
409idempotency_conflictKey reused with a different body.
422amount_too_largeAmount exceeds the per-charge ceiling.
429rate_limitedToo many requests back off and retry.
500internal_errorTransient Aperture fault safe to retry with the same idempotency key.
✍️ Writer's note

The craft in a payments API reference is honesty about failure. It's easy to document the happy path; the value is in the edge cases what a 201 actually commits, why you never gate checkout on a webhook, how idempotency keys make a retry safe, and exactly how the back-off schedule recovers a missed event. I structured this so a developer can integrate from first signed call to reversal recovery without opening a support ticket. Every product, endpoint, and identifier here is invented; no proprietary material is reproduced.
