SEO Documentation

The business problem

A content team publishes steadily, but traffic is flat. SEO lives as tribal knowledge: one writer stuffs keywords, another ignores them, a third writes brilliant articles that no one can find because the title tag is the file name and there's no meta description. Every “SEO review” is a different reviewer applying a different mental rulebook, and the advice they pass on is half folklore from a 2014 blog post.

SEO as a standard, not a superstition

The problem isn't that the team can't write it's that there's no shared, current, written standard for making good content findable. The documentation I built replaces folklore with a repeatable system: every piece is briefed, written, and published against the same checklist, so SEO stops depending on which reviewer happened to look at it.

The three pillars of SEO

The documentation organises everything under three pillars, because almost every real SEO question belongs to one of them. A page ranks when all three hold great content on a page search engines can't crawl is invisible, and a perfectly crawlable page with thin content has nothing to rank.

The three pillars. Rankings come from all three holding together: a technically sound page, well-structured on-page signals, and genuinely useful content that earns links. Strength in one can't rescue failure in another.

PillarOwnsExample questions
TechnicalCrawlability, speed, mobile, indexationCan search engines reach and render this? Is it fast on a phone?
On-pageTitles, headings, metadata, intent matchDoes this page clearly signal what it answers?
Content & authorityUsefulness, depth, trust, linksIs this the best answer to the query? Do others cite it?
🧠

Search intent comes first

The single biggest shift the documentation makes is putting intent before keywords. A keyword is just words; the intent behind it is what the page must satisfy. Matching the wrong intent is the most common reason good content doesn't rank.

IntentThe searcher wantsRight page type
InformationalTo learn somethingGuide, explainer, how-to
NavigationalTo reach a specific siteBrand / product page
CommercialTo compare before buyingComparison, review, “best of”
TransactionalTo do or buy nowProduct, pricing, sign-up
Match the page to the intent, not the keyword

If someone searching “best CRM for small teams” (commercial they want a comparison) lands on a product page that only sells one CRM, they bounce, and the page won't hold a ranking no matter how optimised it is. The documentation makes identifying intent the first step of every brief, before any keyword work.

📝

The content brief SEO before writing

The brief is where SEO is won or lost, because it's far cheaper to plan a page right than to retrofit it. Every piece starts from this template, so the writer knows the target and intent before drafting.

content brief template
Working title:   # draft H1, refined later
            Search intent:   # informational / commercial / etc.
            Primary query:   # the one query this page should win
            Key questions:   # the questions the page must answer
            Audience:        # who is searching, and what do they know?
            Internal links:  # existing pages to link to / from
            Related terms:   # supporting topics to cover naturally
Success metric: # e.g. rank top 5 for the primary query
One page, one primary query

A page that tries to rank for ten queries usually ranks for none. The brief forces a single primary query so the page has a clear job. Related terms are supporting cast, not co-leads covered naturally because they serve the reader, never stuffed to chase a second ranking.

Metadata that earns the click

The title tag and meta description are the page's advertisement in the search results. They don't just describe the page they decide whether anyone clicks it. The documentation gives concrete, current rules.

html · the metadata standard
<title>Primary Query Clear Benefit | Brand</title><!-- ~50-60 chars; primary query near the front 
            --><meta name="description" content="One sentence that answers the query and gives
               a reason to click. ~150 chars, active voice.">               
               <!-- not a ranking factor directly, but drives click-through -->
ElementRuleWhy
Title tag~50–60 chars, primary query near the frontTruncates past ~60; front-loading signals relevance
Meta description~150 chars, active voice, a reason to clickIt's ad copy it earns the click, not the rank
One H1 per pageStates what the page is, matches intentThe clearest on-page topic signal
Descriptive URLShort, words not IDs, hyphen-separatedReadable for humans and engines alike

Structured data for rich results

Structured data (JSON-LD schema) describes a page's meaning to search engines in a machine-readable way, making it eligible for rich results the star ratings, FAQs, and breadcrumbs that make a listing stand out. The documentation gives copy-ready patterns for the common types.

json-ld · FAQ schema, dropped into the page head
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [{
    "@type": "Question",
    "name": "What is a content brief?",
    "acceptedAnswer": {
      "@type": "Answer",
      "text": "A plan that defines a page's query and intent before writing."
    }
  }]
}
Only mark up what's really on the page

Schema must describe content that genuinely appears on the page marking up an FAQ that isn't visible to users is a guidelines violation that can cost the rich result entirely. The documentation is firm on this: structured data describes reality, it doesn't invent it.

✅

The publishing checklist

The standard becomes real at the moment of publishing. This is the gate every page passes before going live.

Before publish
  • Matches the brief's primary query and intent.
  • Title tag and meta description meet the standard.
  • One clear H1; logical heading hierarchy.
  • Descriptive URL; images have alt text.
  • Internal links to and from relevant pages.
  • Structured data added where the type fits.
After publish
  • Submit / confirm the page is indexable.
  • Check it renders and loads fast on mobile.
  • Track the primary query's position over time.
  • Review click-through from the results page.
  • Refresh when the content or intent shifts.

Key learnings

Write for humans first.

Every rule in the standard ultimately serves the reader; the pages that ranked best were the ones most useful to a person, not the ones most stuffed for a crawler.

Intent beats keywords.

Matching the page type to search intent fixed more ranking problems than any amount of keyword tuning.

Plan SEO in the brief.

It's an order of magnitude cheaper to brief a page right than to retrofit SEO after it's written.

📝 Writer's note

SEO documentation ages badly because so much of it chases algorithm rumours. I deliberately wrote this around durable principles crawlability, intent match, genuine usefulness that have held for a decade and will outlast any specific ranking-factor fad.

The framing that made it stick with writers was “write for humans first.” Once the team understood that nearly every SEO rule is really a proxy for being useful and clear to a reader, they stopped seeing SEO as a tax on good writing and started seeing it as a description of it.
