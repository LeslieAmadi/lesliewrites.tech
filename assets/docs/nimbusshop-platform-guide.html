NimbusShop Platform Product System Guide

Product and operational documentation for the NimbusShop Platform, designed for physical game shops with registered devices, player wallets, and real-time reporting.

Overview

Purpose

This document provides a comprehensive guide to the NimbusShop Platform, which is designed for physical game shops. It explains the functionality of the platform, including shop and administrator management, device registration, player wallet operations, and reporting. The guide supports administrators, developers, and QA in understanding, implementing, and using the platform effectively.

Target Audience

  • Developers: For understanding platform workflows and system interactions for implementation and maintenance.
  • Quality Assurance (QA): To design test plans, validate workflows, and ensure operational correctness.
  • Shop Administrators: For day-to-day operations, device management, and player wallet handling.

Scope

In Scope

  • Shop creation and administrator assignment by Super Admins.
  • Device registration, activation, and player interaction.
  • Player wallet operations: deposits, withdrawals, and balance updates.
  • Reporting and analytics, including gameplay and financial records.
  • Account and system support, including password resets and QA environment validation.
  • Demonstration workflows for typical platform usage scenarios.

Out of Scope

  • API integrations outside the standard NimbusShop platform.
  • Development of new game titles or content.
  • Physical hardware maintenance beyond registration and activation within NimbusShop.
  • Advanced customisation or configuration of reporting dashboards beyond gameplay session and transaction overview provided by standard platform features.

Main Features & Functionalities

Pre-Requisites

  • You have access to the platform URL and valid credentials.
  • You are able to set up SSO or password-based MFA authentication.
  • Administrative roles are assigned (Super Admin, Shop Admin).
  • You have access rights to at least one shop.

Logging In

  1. Navigate to [NimbusShop Platform URL].
  2. Click the Sign in button.
  3. Enter credentials (SSO/password) and complete MFA if configured.
  4. If you forgot your password, click “Forgot Password” to receive a reset link via email.

Device Functionalities & Procedures

Example Flow

  1. Register and activate a new device for a shop.
  2. Assign a player wallet and deposit initial funds.
  3. Monitor gameplay and transaction logs in real time.

Screenshot placeholder: Device Registrations page.

Device Activation

The Register Device button redirects the administrator back to the device page, where the activation steps are highlighted.

Clicking Activate This Device enables gameplay on the target machine and resets the current session for the logged-in administrator.

Steps for Device Activation

  1. Navigate to the physical device that needs activation.
  2. Log in as an administrator on that device.
  3. Navigate to the device’s page in NimbusShop.
  4. Click Activate This Device.
  5. An alert is displayed confirming the device was successfully activated and registered.

Device Details

  • Device: e.g. Building (N0) — name and number assigned by the Shop Admin.
  • Inactive / Active: Indicates whether the device is currently active.
  • Back to Location: Returns to the previous shop view.
  • ID: Unique device identifier.
  • Location: Name of the current shop where the device resides.
  • Physical Address: Exact address of the device.
  • Locked: true/false — indicates whether gameplay is paused.
  • Created: Date and time the device was registered.
  • Last Activity: Timestamp of the device’s last activity.
  • Wallets: Current device balance.
  • Edit: Allows the Admin to modify device details.
  • Delete: Allows removal of device registration (subject to permissions).

Device Revenue & Reporting

  • Device Report Date Range: Start and End Date.
  • Update Report: Refreshes device-level KPIs.
  • Current Period GGR: Game Revenue (GGR) for a specific device or building.

Clicking Activate This Device typically displays a pop-up:

  • Cancel: Closes the pop-up and keeps the device inactive.
  • OK: Confirms activation and launches the game lobby.

Screenshot placeholder: Device activation pop-up and game lobby after activation.

Dashboard Functionalities

Shop Statistics (Stats Widget)

The Stats widget displays real-time shop-level operational data, including:

  • Registered Devices: Total number of devices added to the shop.
  • Activated Devices: Subset of registered devices currently active for gameplay.
  • Total Shop Balance: Aggregate of all active device wallet balances within the shop.

These metrics provide administrators with at-a-glance operational status for monitoring shop activity and financial health.

Quick Links Panel

The Quick Links panel provides shortcuts to frequently used functions:

  • Register New Device: Opens the device registration form with required fields like Device ID, Shop Assignment, Activation Status. The system validates the device ID before registration.
  • View Detailed Reports: Redirects to the Location dashboard, where admins can access transaction logs, GGR, deposits, withdrawals, and round history per device and shop. Reports support time, shop, and device filters and can be exported.

Device Money Transfer Module

The Device Money Transfer feature enables administrators to execute financial operations at the device level.

Functional Inputs

  • Choose Device: A dropdown listing all devices assigned to the shop. Admin selects the target device.
  • Transaction Type:
    • Deposit: Credits funds into the selected device wallet.
    • Withdraw: Debits funds from the device wallet and returns cash to the player.
  • Amount: Numeric value; supports multiple currencies based on shop configuration.
  • Process Transfer: Executes the deposit or withdrawal. The system validates transaction limits, device status, and balances, then updates the wallet in real time. On validation failure, an error message is displayed and logged for audit.

Devices Overview

The device overview presents a summarised view of each device, including activation status, locked status, and GGR. This allows administrators to quickly identify underperforming or inactive devices.

Roles & Access

Super Admin

  • Signs into the system to create locations (physical venues).
  • Configures customer credentials to access the game portfolio.
  • Assigns administrators to shops (including multi-shop access).
  • Registers and manages all devices across shops.

Shop Administrator

  • Manages only their assigned shop(s).
  • Registers and activates devices.
  • Manages player wallets (deposits, withdrawals, balance updates).
  • Oversees player gameplay and device interactions.

Players

  • Use registered devices as secure browsers for gameplay.
  • View wallet balances directly on devices.
  • Deposit cash via the shop administrator.
  • Play games and withdraw funds as needed.
  • Optionally lock a device during breaks (unlockable by player or admin).

Shop & Administrator Management

Super Admin Login

Navigate to the login page and sign in using assigned Super Admin credentials.

Shop Creation

From the dashboard, create a new physical location (venue) representing a shop.

Assign Administrators

Allocate shop administrators to specific locations. Multiple shops may be assigned to a single administrator, based on operational requirements.

Admin Access

Shop administrators can log in to view and manage only their assigned shop(s), respecting least-privilege access principles.

Device Registration & Management

Device Onboarding

Shop administrators register devices within their assigned shop through the Devices section.

Activation

Once registered, devices are activated and linked to shop records, enabling gameplay, wallet interactions, and reporting.

Player Interaction

Devices act as access points where players can:

  • View wallet balances.
  • Deposit funds.
  • Play games.
  • Lock/unlock the device during breaks.

Player Wallet & Device Money Transfer

Deposits

Players deposit cash with the shop administrator, who then credits the player’s electronic wallet and/or device balance.

Withdrawals

Administrators process withdrawal requests and pay out cash to players. The system adjusts wallet balances in real time.

Balance & Gameplay

Players can see their real-time balance on the device. All wins, losses, and round outcomes update their wallet automatically.

Security & Audit

Each wallet transaction is logged with timestamps, operator IDs, and device/shop context for regulatory and audit purposes.

Reporting & Analytics

  • Gameplay Records: Logs of rounds played, wins/losses, and device activity.
  • Financial Reports: GGR, deposits, withdrawals, and shop-level financial summaries.
  • Performance Reports: Filters by shop, device, or date range for deeper insight.
  • Compliance & Transparency: Reports suitable for operational review and audits.

Account & System Support

  • Organisation & Super Admin Accounts: Provisioned before client access begins.
  • Password Reset: Users can trigger reset flows via email.
  • Environment: Features can be demonstrated and tested in a dedicated QA environment.

Demonstration Flow (Example Use Case)

The following high-level flow illustrates a typical daily path through the system:

  1. Super Admin logs in and creates a new shop (physical location).
  2. Super Admin assigns one or more Shop Admins to the location.
  3. Shop Admin registers devices for gameplay.
  4. Player arrives, is registered, and deposits cash into a wallet.
  5. Player uses wallet balance to play on a registered device.
  6. Player takes a break and locks the device, then resumes later.
  7. Player requests a withdrawal; Admin processes payout.
  8. Admin reviews revenue, gameplay, and transaction reports.

Daily Task Overview (Shop Admin)

This serves as a quick training and daily operational checklist for shop administrators.

  • Log In: Access the platform with credentials and MFA.
  • Device Management: Register, activate, lock/unlock devices.
  • Player Wallets: Process deposits, withdrawals, and check balances.
  • Reporting: Generate daily compliance and financial reports.

Post-Implementation Checks

  • Each device shows status = Active.
  • All wallet transactions are logged and consistent.
  • End-of-day report totals for deposits, withdrawals, and GGR match physical cash logs within an acceptable variance (e.g. 0.5%).

Best Practices

  • Always verify cash before updating wallet balances.
  • Lock unused devices to prevent misuse.
  • Run daily reports at shop closing.
  • Escalate technical issues promptly to the Super Admin.

Step-by-Step: Player → Wallet → Gameplay → Withdrawal → Reporting

Prerequisites

  • Shop Admin account exists and is logged in.
  • Target device is registered and active in NimbusShop.
  • Cash float is available at the shop for deposits/withdrawals.
  • Network connectivity and NimbusShop services are operational.

1. Player Arrives

Player approaches the shop and requests to play. Staff greet the player and confirm any ID requirements based on shop policy.

2. Admin Registers Wallet

Admin navigates to Players / Wallets → Create New Player / Register Wallet. The system creates a player profile and electronic wallet with a unique ID.

3. Player Deposits Cash

Admin goes to the player profile, selects Wallet > Deposit, enters amount, and confirms cash receipt. The system records the deposit and queues it for immediate processing.

4. Wallet Updated

Wallet balance is updated in real time; transaction ID is generated and logged. Player can see the new balance on the device.

5. Player Plays on Device

Player selects a game using device credits. The device debits the wallet for each round, and the session is tracked in NimbusShop.

6. Balance Displayed in Real Time

After each round, the balance is updated on both the device UI and the admin dashboards. Transaction logs capture debits and credits per round.

7. Break Taken → Device Locked

Player or Admin chooses Lock Device. The system pauses gameplay, hides balance, and logs a lock event.

8. Device Unlocked

Player unlocks with a PIN, or Admin unlocks via the dashboard. The system verifies identity and resumes session where appropriate.

9. Player Requests Withdrawal

Player requests a withdrawal; Admin reviews balance and pending holds in the wallet view and prepares to process.

10. Admin Processes Withdrawal

Admin selects Wallet > Withdraw, enters amount, and confirms payout. System deducts funds, logs the transaction, and the player receives cash.

11. Wallet Updated (Post-Withdrawal)

Balance is updated, and an audit trail (operator ID, time, device, shop) is persisted for compliance.

12. Admin Runs End-of-Day Reports

Admin generates a daily report summarising deposits, withdrawals, GGR, and device activity. Totals are reconciled against physical cash.

Common Errors & Quick Fixes

  • Deposit not recorded: Retry transaction; if risk of duplication exists, escalate to Super Admin for reconciliation.
  • Device unresponsive: Deactivate device in dashboard, reassign player to another device, and log a support ticket.
  • Balance mismatch: Export transaction history, reconcile deposits/withdrawals, and check for pending holds or errors.
  • Unauthorised unlock attempt: Block action, confirm player identity, and escalate if suspicious.

Audit & Logging Notes

  • Each deposit/withdrawal and lock/unlock event must create a timestamped audit record.
  • Records must include operator ID and device/shop context.
  • End-of-day reports and logs should be retained per regulatory retention policy.
