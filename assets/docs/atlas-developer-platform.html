The business problem

A developer evaluating an API gives it about ten minutes. If they can't authenticate and get one successful response in that window, they leave and an API nobody can adopt has no value no matter how good the underlying platform is. Atlas had a capable API and documentation that read like a database dump: every field listed, no path through it, no runnable example, no answer to “what do I actually type first?”

Documentation is the product's front door

For a developer platform, the docs are the product experience nobody calls a salesperson, they read the reference and try a call. I rebuilt the Atlas reference around time-to-first-success: authenticate, make one call, see one response, on the same screen. The structure below is what turns an evaluator into an integrator.

Authentication

Atlas authenticates with a secret API key sent as a Bearer token on every request. Keys are environment-scoped a sk_test_ key hits test data, a sk_live_ key hits production so you can build safely before going live.

⚠ Keep secret keys server-side

A secret key can move money and read data. Never ship it in client-side code or commit it to source control; load it from an environment variable.

Authorization header
curl https://api.atlas.dev/v1/resources \
-H "Authorization: Bearer sk_test_4eC39Hq..."
⚡

Quickstart — your first call

The fastest path to a successful response: create a resource. The tabs on the right show the same call in three languages pick yours and run it. A 201 Created with a resource id means you're integrated.

Every Atlas object has a stable, prefixed id (here res_), a created timestamp, and a livemode flag so you always know which environment produced it.

POST /v1/resources
curl https://api.atlas.dev/v1/resources \
-H "Authorization: Bearer sk_test_4eC39Hq..." \
-d name="My first resource" \
-d type=standard
Node — atlas-node
const atlas = require("atlas-node")(process.env.ATLAS_KEY);

const resource = await atlas.resources.create({
name: "My first resource",
type: "standard",
});
Python — atlas
import atlas
atlas.api_key = os.environ["ATLAS_KEY"]

resource = atlas.Resource.create(
name="My first resource",
type="standard",
)
200 — response
{
"id": "res_1Nl8x2eZvKYlo2C",
"object": "resource",
"name": "My first resource",
"type": "standard",
"status": "active",
"created": 1718000000,
"livemode": false
}

The resource endpoints

Atlas follows REST conventions: the same resource path supports create, retrieve, update, list, and delete through standard HTTP verbs.

MethodPathDescription
POST/v1/resourcesCreate a resource
GET/v1/resources/:idRetrieve a resource
POST/v1/resources/:idUpdate a resource
GET/v1/resourcesList resources
DELETE/v1/resources/:idDelete a resource

Parameters — create a resource

name requiredA human-readable name for the resource.
type requiredOne of standard or priority.
metadata optionalA set of key-value pairs you can attach for your own use.
GET /v1/resources/res_1Nl8x2
curl https://api.atlas.dev/v1/resources/res_1Nl8x2 \
-H "Authorization: Bearer sk_test_4eC39Hq..."
♻️

Idempotency

Networks fail mid-request. To safely retry a create without risking a duplicate, send an Idempotency-Key header with a unique value. Atlas remembers the first response for that key for 24 hours and replays it on any retry, so the resource is created exactly once.

Use it on every POST that creates or charges

A retried create without an idempotency key is the classic source of duplicate records. One header eliminates the entire class of bug.

safe retry
curl https://api.atlas.dev/v1/resources \
-H "Authorization: Bearer sk_test_4eC39Hq..." \
-H "Idempotency-Key: a1b2c3-unique" \
-d name="My first resource" \
-d type=standard
📄

Pagination

List endpoints are cursor-paginated. Pass limit (1–100) and, to get the next page, starting_after set to the last id you received. The response's has_more tells you when to stop.

GET /v1/resources?limit=2
{
"object": "list",
"has_more": true,
"data": [
{ "id": "res_1Nl8x2" },
{ "id": "res_1Nl8y7" }
]
}

Errors

Atlas uses conventional HTTP status codes and returns a structured error body. This is the page developers visit most, so it maps every code to a cause and a fix.

CodeMeansDo
400Bad request a parameter is malformedCheck the message field
401No or invalid API keyVerify the Authorization header
404Resource doesn't existCheck the id
409Idempotency key reused with a different bodyUse a fresh key
429Rate limitedBack off and retry with jitter
500Atlas errorRetry; if it persists, contact support
error body
{
"error": {
"type": "invalid_request_error",
"param": "type",
"message": "type must be standard or priority"
}
}
🔔

Webhooks

Rather than polling, register a webhook endpoint and Atlas POSTs an event to it when something happens for example resource.updated. Always verify the Atlas-Signature header before trusting a payload, so you know the event genuinely came from Atlas.

⚠ Verify the signature, then respond 2xx fast

Acknowledge with a 2xx immediately and process asynchronously slow webhook handlers cause Atlas to retry and your events to back up.

event payload
{
"id": "evt_1Nl9a2",
"type": "resource.updated",
"data": {
"object": { "id": "res_1Nl8x2" }
}
}

Key learnings

Optimise for time-to-first-success.

Every structural choice served one goal: get the developer to a successful response as fast as possible. That metric drove the whole reference.

Show the call, don't describe it.

Pairing each concept with a runnable example on the same screen did more than any amount of prose.

The error page is a feature.

Mapping every code to a cause and a fix turns the most frustrating moment in an integration into a self-service one.

📝 Writer's note

I modelled this reference on the documentation pattern the industry treats as the benchmark prose paired with runnable code in a two-column layout because for a developer platform, the docs are the product. The patterns are borrowed; the visual language, palette, and structure are this site's own.

The deeper craft was in what most API docs omit: idempotency, pagination, and a real error reference. Those aren't glamorous, but they're exactly what separates documentation a developer can demo with from documentation they can build a business on. Writing them as first-class sections is the difference between an API that looks usable and one that is.
