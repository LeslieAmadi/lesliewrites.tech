Helix Play Operator API

A seamless-wallet integration between the Helix remote game server and an operator's lobby and wallet: signed play sessions, real-time wallet hooks for every round, and reversal-based recovery when a hook can't be confirmed. Written for an operator's engineers to integrate safely against real money.

How the integration works

The operator owns the player's money. Helix owns the game logic. They meet at two surfaces:

  • The operator opens a play session by calling Helix, and redirects the player to the returned game client.
  • As the player plays, Helix calls the operator's wallet hook for every round, asking it to move money and report the resulting balance.

Roles & terms

TermMeaning
Play sessionOne player, one game, one currency. Created by the operator; identified by a session_id.
RoundA single interaction that moves money: a stake out of the wallet and/or a return into it.
Feature cycleA chain of related rounds sharing one root free rounds, gambles, and similar follow-on play.
Wallet hookThe operator endpoint Helix calls per round. The operator's reply decides whether the round stands.
Reversal hookThe operator endpoint Helix calls to unwind a round that could not be confirmed.
ledger_refThe globally unique id Helix assigns to a round the key the operator stores to reconcile reversals.

Request signing

Helix uses HTTP message signatures. Both sides sign: the operator signs requests to Helix, and Helix signs every hook it sends to the operator. A request carries a Digest of its body and a Signature header naming exactly which fields were signed.

HeaderDescription
Helix-DateRFC 1123 timestamp of signing. Requests outside a ±60s window are rejected.
DigestSHA-384=<base64> of the raw body. For empty bodies, the digest of the empty string.
SignatureNames the keyId, algorithm, the ordered headers covered, and the signature itself.

Building the signing string

Take the fields listed in headers, in order. Emit each as name: value on its own line, joined by newlines. The pseudo-field (request-target) is the lowercase method, a space, then the path. HMAC-SHA384 that string with the operator secret; base64 the result.

Signing string
(request-target): post /2024-09/sessions
helix-date: Wed, 11 Dec 2025 14:00:15 GMT
digest: SHA-384=Yk9u…q2A=

signature = base64( HMAC_SHA384( secret, signing_string ) )
Worked example signing a session create

algorithmhmac-sha384

Signature headerkeyId="op_live_42",algorithm="hmac-sha384",headers="(request-target) helix-date digest",signature="QmF…9Zg="
⚠ Verify hooks before you trust them

Always reject a hook whose Digest doesn't match a freshly computed hash of the received body, then verify the signature over the named headers using a constant-time compare. Order matters: the headers list defines the exact lines and their sequence.

ℹ Launching the client

Redirect the player to client_url by HTTP redirect, an <iframe>, or a native web view. If you omit return_url, make sure your shell can return the player to the lobby when the session ends.

The wallet hook

For every round Helix POSTs to your wallet_hook. The request is signed. Your job: apply the money movement and reply with the player's new balance or decline with a reason. Reply 200 within 10 seconds; anything else is a failure and triggers recovery.

session_idstring requiredThe session this round belongs to.

stakeinteger requiredMinor units to remove from the wallet (may be 0).
returninteger requiredMinor units to add to the wallet (may be 0).

Confirming a round

Compute the new balance as balance − stake + return, guarding against a negative result before applying the return. Reply with the authoritative balance.

200 OK
HTTP/1.1 200 OK
Content-Type: application/json

{ "balance": 49750, "ledger_ref": "lr_4Np7Kq2" }
⚠ Persist in the same transaction

Persist the round against its ledger_ref in the same transaction that moves the money. If you can't look up a round by ledger_ref later, you cannot safely process its reversal.

Declining a round

To reject the round, still reply 200 but with a decline object. Helix discards the round and shows the player a message.

200 OK declined
HTTP/1.1 200 OK
Content-Type: application/json

{
  "decline": {
    "reason": "insufficient_funds",
    "message": "You don't have enough balance for this stake."
  }
}
reasonMeaning
insufficient_fundsStake exceeds balance. No message required.
limit_reachedA deposit, loss, or wager limit stopped play.
self_excludedThe player is excluded from real-money play.
session_expiredThe session is no longer valid on the operator side.
customOperator-defined; supply your own message.

Failure & recovery

A wallet hook can fail technically (timeout, 5xx, unreachable, unparseable) or simply not arrive. Helix offers two recovery modes per operator choose the one that matches what your wallet can do.

compensate (default)

The round is unwound: Helix calls your reversal_hook, and the player may resume once it succeeds. Best for operators whose ledger can reverse an applied movement.

settle

The round is held and the wallet hook is retried until it succeeds; play is locked meanwhile. The reversal hook is never called. Best for operators who cannot reverse and prefer eventual consistency. Requires idempotent hooks.

The reversal hook

In compensate mode Helix POSTs the original round body to your reversal_hook, keyed by the same ledger_ref. Undo any movement you made for that round and reply with the corrected balance.

POST /helix/reversal
POST /helix/reversal HTTP/1.1
Helix-Date: Wed, 11 Dec 2025 14:01:20 GMT
Digest: SHA-384=Re8v…0aQ=
Signature: keyId="helix_rgs",algorithm="hmac-sha384", …

{ "ledger_ref": "lr_4Np7Kq2", "session_id": "ses_0Ab9Qx", "round": 12,
  "stake": 250, "return": 0 }
ℹ If you never saw the original

If the original wallet hook was lost in transit, there is nothing to undo. Make no change and reply with the current balance and the same ledger_ref. Reversals must be safe to receive more than once.

Retry schedule

Both hooks share one back-off. The first two attempts fire immediately; if they fail, play locks for that game and currency and Helix retries in the background.

AttemptDelayCumulative
r0 · r1immediate0s
r25s5s
r330s35s
r45m~5m
r51h~1h
r66h~7h
r7 (final)18h~25h

ℹ Bonus stakes are your call

Whether bonus stakes draw from the real wallet is the operator's decision. If you choose not to deduct bonus stakes, treat the hook's stake as informational and reply with the unchanged balance.

Stopping a promotion

POST/2024-09/promotions/{promotion_id}/stop

Send { "grace_seconds": 900 } to let players who already started keep playing for a window, or 0 to end it for everyone at once.

Error reference

Errors returned to the operator (session create, promotions) use a stable code with a human message. These are distinct from wallet-hook decline.reason values, which the operator sends to Helix.

HTTPcodeMeaning
401signature_invalidSignature did not verify over the named headers.
401digest_mismatchBody digest did not match the request body.
401date_skewHelix-Date outside the ±60s window.
404game_unknownNo game with that game_code.
404session_unknownNo session with that id.
409session_lockedA pending reversal or settle is blocking the session.
422currency_unsupportedThe game does not offer the requested currency.
422hook_urls_identicalWallet and reversal hooks must differ.
422stake_not_a_bet_stepstake_per_round is not a valid bet step.
429rate_limitedToo many requests; honour Retry-After.
500rgs_errorTransient server fault safe to retry.
✍️ Writer's note

A seamless-wallet RGS integration is unforgiving: real money moves on every spin, and the hard part is the moment a hook doesn't get confirmed. I built this reference around that reality the ledger_ref that makes a round reconcilable, the two recovery modes and when each fits, the reversal that must be safe to receive twice, and the exact back-off that locks play until the ledger is consistent again. The compliance-adjacent decline reasons (limits, self-exclusion) are first-class, not footnotes. Every game, endpoint, and identifier here is invented; no proprietary material is reproduced.
