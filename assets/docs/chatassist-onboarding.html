ChatAssist Onboarding Guide

The companion to the ChatAssist architecture: a step-by-step path that takes a developer from an empty project to a live, grounded support assistant. Prerequisites, five concrete setup steps with real code, connecting a knowledge base, testing, a go-live checklist, and the troubleshooting that answers “why isn't it working?” before they have to ask.

The business problem

A powerful assistant nobody can set up delivers nothing. ChatAssist's architecture is sound, but a developer dropped into it cold faces a wall of questions: what do I need first, where does my knowledge base go, how do I test it, and how do I know it's safe to ship? Without a guided path, every integration becomes a support ticket, and a slow first experience sours the whole product.

Onboarding is where adoption is won or lost

The architecture doc explains how ChatAssist works; this guide gets a developer to a working assistant. It's deliberately built around one promise a grounded answer in about fifteen minutes because the first fifteen minutes decide whether a developer commits to the integration or abandons it.

The onboarding path

Five steps take you from nothing to a grounded answer. The diagram shows the whole journey; the steps below give the code.

The five-step onboarding path: authenticate, load and index your knowledge base, ask a question, test that answers are grounded, then go live against the checklist.

1

Authenticate

Install the SDK and initialise it with your key from an environment variable.

Node
const chatassist = require("chatassist")(process.env.CHATASSIST_KEY);
2

Load & index your knowledge base

Upload your content. ChatAssist chunks, embeds, and indexes it so it can be retrieved at query time. This is the grounding source the assistant answers only from here.

Node index KB documents
await chatassist.knowledge.upload({
  source: "./help-center/",   // FAQs, docs, articles
  language: "auto",           // EN + DE handled together
});
3

Ask a question

Send a user message. The response includes the grounded answer and the sources it used, so you can verify it.

Node query
const reply = await chatassist.ask({
  message: "How do I reset my password?",
});
console.log(reply.answer);   // the grounded answer
console.log(reply.sources);  // passages it used
4

Test grounding & escalation

Ask something not in your knowledge base. A correctly configured assistant won't invent an answer it escalates. That negative test matters more than the happy path.

Node the test that matters
const r = await chatassist.ask({ message: "What's the weather today?" });
// expect: r.escalated === true, not a made-up answer
5

Go live

Swap the test key for a live key, point ChatAssist at your real escalation channel, and run the go-live checklist below before exposing it to customers.

🚀

Go-live readiness gate

“It answered my test question” is not the same as “it's ready for customers.” This gate is the difference.

The readiness gate. An integration only goes live when grounding is verified, escalation is wired to a real channel, the live key is in place, and logging is on — the same all-criteria-must-pass discipline used across the platform's other gates.

Ready to ship
  • Grounded answers verified against known questions.
  • Out-of-scope questions escalate, don't fabricate.
  • Escalation routes to a real human channel.
  • Live API key in place, secret kept server-side.
  • Logging and observability enabled.
Not yet
  • Answers without sources, or sources that don't match.
  • The bot answers questions outside the knowledge base.
  • Escalation goes nowhere.
  • Test key still in use, or key exposed client-side.
  • No way to see what the assistant is doing.
🔧

Troubleshooting

The questions every first integration hits answered here so they don't become support tickets.

SymptomLikely cause & fix
“The bot says it doesn't know”Knowledge base empty or not indexed yet confirm step 2 completed and the content covers the question.
“Answers have no sources”Querying before indexing finished, or grounding disabled re-run indexing and check the response includes sources.
“It answers off-topic questions”Scope guardrail not enabled turn on the scope limiter so it stays within the KB.
401 UnauthorizedMissing or wrong key check the environment variable and that you're using the right environment's key.
“German answers are weak”KB is English-only multilingual retrieval needs German content indexed too.

Key learnings

Promise a time, then honour it.

“A grounded answer in ~15 minutes” gave the whole guide a target and the reader a reason to start.

The negative test is the real test.

Making “ask something not in the KB and confirm it escalates” a required step is what proves the assistant is safe.

Pre-answer the support tickets.

A troubleshooting table built from the predictable first-integration failures deflects most onboarding tickets before they're filed.

📝 Writer's note

An onboarding guide is a different craft from a reference: a reference answers “what does this do?”, onboarding answers “what do I do next?” I wrote every step as an imperative with code attached, so the developer is always doing, never just reading.

The decision I'm proudest of is making the negative test ask something the assistant shouldn't know a first-class step rather than an afterthought. For a RAG assistant, proving it declines gracefully is more important than proving it answers, and putting that in the happy path is what turns a quick demo into a safe production integration.
