The business problem

A marketing campaign goes live at 9 a.m. Within ten minutes, traffic to the recommendation model doubles. Latency climbs from 180 ms to 2 seconds. User-facing features that depend on the model start timing out. The on-call engineer is paged but there is no runbook, no documented scaling behaviour, and no clear answer to a simple question: is this the platform failing, or the model, or the integration?

Helios was built so that question never has to be asked in a panic. It is a shared model-serving platform hosting many teams' models in production real-time APIs, overnight batch jobs, and scheduled inference. But a platform shared by many teams only works if every team understands how it behaves under load, how it fails, and how to operate it safely. That understanding is documentation. That was my job.

The documentation goal

Produce documentation that lets any ML team deploy a model to Helios, understand exactly how it will behave under real traffic, monitor it against clear SLOs, and follow a tested runbook when something goes wrong without needing a platform engineer on the call. The platform is multi-tenant; the documentation had to serve teams who had never spoken to each other.

System architecture

Helios separates the control plane (configuration, model registry, deployments, canary control) from the data plane (the actual inference traffic). This separation is the single most important architectural fact for any team using the platform: changes to configuration never directly touch live traffic, and traffic spikes never destabilise the registry.

1
Request enters via API Gateway

Authentication, rate limiting, schema validation at the edge

2
Router selects model version

By model ID + version pin, traffic-split rules, region, and pod health

3
Inference pod serves the request

GPU or CPU pod, warmed into memory before being marked Ready

4
Logging & metrics captured

trace_id and model_version logged; latency and error metrics exported

5
Response returned to client

Unhealthy pods are automatically removed from rotation to avoid cascading failure

Why the two-plane split matters to you

When you deploy a model, you interact with the control plane registering versions, setting traffic splits, configuring canaries. Live user traffic flows through the data plane. A misconfigured deployment is caught in the control plane before it ever sees real traffic. This is the platform's core safety property.

Diagram 1 of 3 Platform architecture. The control plane (right) manages the registry, deployments, and canary control; the data plane (left) carries live inference traffic. A misconfigured deployment is caught in the control plane before it reaches a single real request.

How routing actually works

"The router selects a model version" is the summary; here is the mechanism. The router makes its decision on four inputs, evaluated in order:

Routing inputMechanismExample
Version pinExplicit model ID + version requested by the caller, if anyFraudModel@2.3
Weighted traffic splitPercentage allocation across versions, set in the control plane90% v2.3, 10% v3.0-canary
Region / tenant affinityRequests pinned to the nearest healthy region; tenant isolation respectedeu-west caller → eu-west pods
Pod healthUnhealthy pods are removed from rotation automatically; traffic shifts to healthy replicasFailing pod drained, not served

Regional failover is automatic: if an entire region's pods are unhealthy, the router fails over to the next nearest region rather than returning errors. This is the behaviour that prevents a single-zone incident from becoming a platform-wide outage.

Serving modes

Helios offers three serving modes. Choosing the right one is the first decision any team makes.

Online / real-time
Latency-sensitive

Personalised recommendations, fraud checks, in-product risk alerts. P95 under 250 ms. Steady baseline with spikes. Strict SLOs and graceful degradation if dependencies fail.

Batch / async
High volume

Nightly credit-risk scoring for millions of accounts, weekly churn predictions, backfills after schema changes. Queue-based, tolerant of retries, backpressure, and longer runtimes.

Shadow / replay
Validation

Duplicates live requests to a candidate model without affecting user-visible behaviour. Real-world load testing, side-by-side comparison, edge-case validation before promotion.

Model registry & promotion workflow

Every model deployed through Helios must be registered in a central registry tracking its ID, version, artifact URI, input/output schema, owner, and status. The registry controls which version receives live traffic and who is allowed to roll back during an incident.

Promotion workflow

  1. Register the model with metadata and linked evaluation results.
  2. Deploy to staging for integration tests.
  3. Shadow or canary in production typically 5–10% of traffic.
  4. Track dashboards latency, error rate, and business KPIs side by side.
  5. Shift to 100% after sign-off.
  6. Keep the old version as hot standby for fast rollback.
⚠ The hot-standby rule

The previous production version is never decommissioned immediately. It is kept warm and ready so that a rollback during an incident takes seconds, not minutes. This is the difference between a 30-second blip and a 30-minute outage. Document which version is the current standby in every deployment record.

Diagram 2 of 3 Promotion and rollback flow. Every new version passes through a canary gate before full rollout. The decision is metric-driven: healthy traffic rolls out to 100%; degraded traffic rolls back to the hot standby and the fix is re-tested from staging.

Autoscaling & the campaign-spike scenario

Helios supports horizontal autoscaling (more pods) and vertical scaling (bigger instances). Scaling triggers on P95 latency, CPU/GPU utilisation, queue depth, and custom business metrics. Here is the scenario from the opening of this page, documented as expected behaviour:

Real scenario — campaign traffic doubles in 10 minutes
1
Autoscaler detects sustained CPU and latency increase above threshold.
2
Spins up additional pods for example, from 8 to 20 replicas.
3
New pods warm the model into memory before serving any traffic.
4
Rate limiter continues to protect upstream services from overload.
5
Dashboards show a temporary latency bump, then stabilisation.

If scaling cannot keep up for example, GPU capacity is exhausted Helios degrades gracefully:

  • Returns cached responses where it is safe to do so.
  • Falls back to a cheaper, lighter model.
  • Fails "open" or "closed" depending on the risk profile of the product.

Logging & tracing

Every request carries a trace_id and model_version. Logs capture input metadata never raw PII alongside timing and a result summary. Distributed tracing shows the full gateway → router → pod path, so when latency climbs you can see exactly which hop is slow.

SLI definitions & error budgets

An SLO without an SLI is an aspiration, not a target. Each objective below is backed by a precise service-level indicator the exact ratio measured plus a measurement window and the resulting error budget the team is allowed to spend before remediation becomes mandatory.

SLOSLI (what is measured)TargetWindowError budget
AvailabilitySuccessful inference requests ÷ total inference requests99.9%30 days43.2 min/month
LatencyRequests served under 250 ms ÷ total requests95% < 250 ms30 days5% may exceed
Error rate5xx responses ÷ total responses< 0.5%Rolling 1 h~18 errors per 3,600 req
Correctness (canary)Predictions within precision tolerance ÷ scored predictionsPrecision ≥ 92%Per canary window>5% drop → auto-rollback
⚠ How the error budget is used

The 43.2-minute monthly availability budget is a spending account. If a team burns it on a risky deployment, feature work pauses and reliability work takes priority until the budget recovers. This converts "be reliable" from a vague instruction into a number every team can see on a dashboard. When the budget is healthy, teams ship faster; when it is exhausted, the policy makes the trade-off automatic rather than a debate.

Failure modes & incident response

The documentation covers the four most common failure modes, each with a tested runbook:

Failure modeSymptomFirst response
Model regressionNew version performs worse on key metrics (precision drops, false positives rise)Auto-rollback to previous version per policy; page owning team
Latency spikeP95 climbs above SLO; downstream dependency is slowCheck tracing for the slow hop; scale out if utilisation-bound
Capacity exhaustionNot enough GPU/CPU; autoscaler cannot keep upEngage graceful degradation; escalate capacity request
Schema mismatchClient sends fields the model does not expectReject at gateway with clear 400; notify integrating team
Executable runbook model regression (SEV-2)
SeveritySEV-2 — degraded prediction quality, no full outage
Symptoms
  • Precision < 92% on the canary cohort.
  • False-positive rate > 3%.
  • Business KPI (e.g. recommendation CTR) degrading.
Verify
shell · confirm the regression before acting
kubectl get deployment fraud-model o wide      # confirm canary version is live
promtool query instant 'precision:fraud_model:5m' # current precision
# open Grafana → "Model Quality" dashboard, compare canary vs baseline
Decision tree
  • If precision drops > 5% vs baseline → roll back immediately (below).
  • If precision drops < 5% and stable → hold canary, continue monitoring, notify owning team.
  • If business KPI is affected at any drop → roll back regardless of precision delta.
Roll back
shell · roll back to hot standby
helm rollback fraud-model            # revert to previous (hot standby) release
                kubectl rollout status deploy/fraud-model
                 promtool query instant 'precision:fraud_model:5m' # verify recovery
Success criteria
  • Precision restored to baseline (≥ 92%).
  • Error rate back under 0.5%.
  • Business KPI stable for 15 minutes.
  • Post-incident review scheduled; logs, dashboards, and root cause linked.

Key learnings

Document behaviour under load, not just the happy path

The most valuable documentation for a platform is not "here is how to call it" it is "here is exactly what happens when traffic doubles, when a model regresses, when GPUs run out." Engineers do not panic when they have already read the expected behaviour. The campaign-spike scenario was the single most-referenced page in the entire set.

A runbook is only real once it has been tested

I wrote every incident playbook with the SRE team and then we walked through each one against the staging environment before publishing. A runbook that has never been executed is a guess. The "rollback plan tested, not just written" checklist item came directly from this lesson.

Shared infrastructure needs documentation written for strangers

On a multi-tenant platform, the person reading your documentation may be on a team you have never met, working on a model you know nothing about. That forces a discipline: no assumed context, no tribal knowledge, every term defined. It makes the documentation better for everyone, including the teams you do know.
