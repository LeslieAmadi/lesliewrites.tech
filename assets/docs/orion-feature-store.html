Orion Feature Store

A feature store is the shared data layer that lets machine-learning teams compute a feature once and use it everywhere in training and in production without the two drifting apart. I documented the full system: offline and online stores, the feature registry, point-in-time correctness, materialisation, and the training/serving skew problem that a feature store exists to solve.

The business problem

A data scientist trains a fraud model. One of its strongest features is average transaction amount over the last 7 days. In training, that feature is computed with a tidy SQL query over a historical table. The model scores well and ships.

In production, a different team re-implements that same feature in the serving code this time in a streaming job, under time pressure, with a subtly different window boundary and a different way of handling customers with no recent transactions. The number the model sees in production is now not the number it was trained on. Performance quietly erodes, and nobody can see why, because both the training query and the serving code look correct in isolation.

This is training/serving skew and it is the problem Orion exists to solve

When a feature is computed one way for training and another way for serving, the model is effectively being asked a different question in production than the one it learned to answer. A feature store removes the second implementation entirely: the feature is defined once, and both training and serving read from that single definition. The documentation is what makes that discipline usable across every team rather than a rule people forget under deadline.

Architecture

The core idea is one transformation feeding two stores. A feature is defined and computed once, then materialised into an offline store tuned for large historical reads (training) and an online store tuned for low-latency single-row reads (serving). A central registry owns the definitions so both paths stay identical.

Diagram 1 of 3 Architecture. One transformation feeds two stores. The offline store serves high-volume historical reads for training; the online store serves low-latency single-row reads for production. The registry owns every definition so both paths compute the same feature the same way.

ComponentOptimised forRead pattern
Offline storeLarge historical scans across many rows and datesBatch, high throughput
Online storeSingle-entity lookups under a latency budgetPoint read, low latency
RegistryDefinitions, ownership, versions, lineageMetadata
MaterialisationMoving computed features into each store on scheduleBatch + streaming

Training/serving skew the core failure

Skew happens when the feature used in training is not identical to the feature used in serving. The model learned a relationship using one set of numbers, then meets different numbers in production. The diagram shows the two worlds and where they diverge.

Diagram 2 of 3 Training/serving skew. Two separate implementations of the same feature drift apart at the edges window boundaries, null handling, rounding. A feature store removes the second implementation: both paths read one definition.

⚠ Why “just be careful” doesn't work

Skew is rarely a big, obvious bug. It hides in edge cases: how a 7-day window treats the current partial day, what happens for a brand-new customer with no history, whether amounts are rounded before or after averaging. Two engineers can each write correct-looking code and still disagree on these. The only durable fix is a single definition both paths share which is what a feature store enforces structurally rather than by discipline.

⏲️

Point-in-time correctness

This is the concept that separates a real feature store from a glorified cache, and the one the documentation spends the most care on. When you build a training set, each training row must see features as they were at that row's moment in time never values that only became known later. Joining in a feature value from after the label event leaks the future into the model and produces training scores that collapse in production.

Diagram 3 of 3 Point-in-time correctness. For a training row anchored at the label event, the feature store joins the value that was known before that moment and excludes anything known only afterward. This is what prevents label leakage.

The tell-tale symptom

If an offline model looks almost too good near-perfect on the validation set the first thing to suspect is a point-in-time violation: a feature that quietly encodes the outcome. The documentation teaches reviewers to treat suspiciously high offline scores as a leakage signal, not a success, which connects this page directly to the NeuralFlow evaluation work.

Materialisation & freshness

Materialisation is how a computed feature actually lands in each store. The documentation makes the freshness contract explicit, because a serving team needs to know how old the value behind a feature can be before it stops being trustworthy.

Feature typeMaterialised byTypical freshness
Batch (e.g. 30-day averages)Scheduled batch jobHours refreshed daily or hourly
Streaming (e.g. last-5-minute counts)Streaming pipelineSeconds to minutes
On-demand (e.g. request-time ratios)Computed at serving timeReal-time, no staleness
Freshness is a contract, not a detail

A feature that is hours stale may be perfectly fine for one model and dangerous for another. By documenting freshness as an explicit SLA per feature, Orion lets a consuming team decide whether a feature is safe for their use case instead of discovering the staleness in production.

Feature authoring guide

The whole value of a feature store collapses if features are authored inconsistently. This is the guide an engineer follows to define a feature once, correctly, for both paths.

Runbook define and register a new feature
1 · Define the feature once
python · one definition, both paths use it
@feature(entity="customer", freshness="1h") def avg_txn_amount_7d(txns):
                # single source of truth for training AND serving
                window = last_n_days(txns, 7) return mean(window.amount) if window else 0.0
2 · Register it
shell · publish to the registry with an owner
orion feature register avg_txn_amount_7d --owner fraud-team 
              # now discoverable and reusable by any team
3 · Build a training set (point-in-time correct)
python · the store enforces no-future-leakage
training_df = orion.get_historical_features(entity_df=labels,            
                # each row has its own event timestamp features=["avg_txn_amount_7d"],)  
                # joins values as-of each label's timestamp
4 · Read at serving time
python · identical feature, low-latency read
features = orion.get_online_features(entity={"customer": cust_id}, features=["avg_txn_amount_7d"],)  
                # same definition as training → no skew
Checklist before publishing
  • Feature has a named owner and a freshness SLA.
  • Null / cold-start behaviour is defined explicitly (here, new customers return 0.0).
  • Training reads use point-in-time joins, never a plain join on latest value.
🛡️

Discovery, ownership & governance

A feature store only pays off if teams reuse features instead of quietly rebuilding them. That requires features to be discoverable and trustworthy which is a governance and documentation problem as much as a technical one.

ConcernRule documented
OwnershipEvery feature has a named owning team responsible for its correctness.
VersioningChanging a feature's logic creates a new version; the old one is retained for models that depend on it.
DeprecationA feature can't be deleted while a live model consumes it; deprecation is announced with a migration window.
DiscoveryThe registry is searchable with descriptions, owners, freshness, and lineage so teams find before they build.
Reuse is the real return on a feature store

The skew fix is what justifies a feature store technically, but reuse is what justifies it to a business. When a well-documented “customer 7-day spend” feature already exists, owned and freshness-guaranteed, the next three teams that need it discover it instead of writing three more subtly different versions which is three more chances for skew avoided.

Key learnings

The hardest concept deserves the most documentation.

Point-in-time correctness is where teams trip, so it earned the most worked examples. Documentation effort should follow conceptual difficulty, not section length conventions.

A feature store is a contract, not a database.

The value isn't storage it's the promise that training and serving see the same number. Framing every page around that contract made the docs cohere.

Governance is what makes reuse safe.

Discovery without ownership and versioning just spreads risk faster. The governance model was what turned reuse from a hazard into the platform's biggest win.

📝 Writer's note

Feature stores are easy to document badly as a list of API calls. I built this around the one problem the whole system exists to solve, training/serving skew, because an engineer who understands why the single-definition rule matters will follow it under deadline, while one who only memorised the API will quietly re-implement a feature when it's faster.

Point-in-time correctness got the most care because it's both the most powerful feature and the easiest to get catastrophically wrong. Connecting it to the “too-good offline score” symptom links this page to the NeuralFlow evaluation work, so the portfolio reads as one coherent platform rather than three unrelated documents.
