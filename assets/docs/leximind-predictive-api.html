The LexiMind Health Intelligence Model (HIM-2.0) provides clinically oriented risk estimates for chronic conditions using structured patient data. The service is designed to plug into electronic health record (EHR) systems, decision-support dashboards, and analytics pipelines as a supporting signal never as an autonomous diagnostic tool.

Who this page is for. Engineers, solution architects, and technical stakeholders responsible for integrating the LexiMind Health Intelligence API into clinical systems. It covers architecture, endpoints, request/response contracts, model characteristics, performance expectations, error handling, and guidance for safe use.

1. High-level overview

LexiMind’s predictive diagnosis capability is delivered as a stateless JSON-over-HTTPS API. Clients submit structured patient data (age, vitals, lab values, risk factors) and receive:

  • Per-condition risk scores in the range [0, 1].
  • An overall risk category (low, medium, high).
  • Human-interpretable feature contributions to support clinical interpretation.

2. Environments & authentication

2.1 Base URL

The v1 API is exposed under:

  • Production: https://api.leximind.ai/v1

If your organisation uses a dedicated staging environment, a similar pattern (for example https://staging.api.leximind.ai/v1) is recommended and can be provisioned per customer.

2.2 Authentication

Authentication is handled via API keys. Every request must include a bearer token: 

Authorization: Bearer YOUR_API_KEY
  • Keys are scoped per organisation and environment.
  • Store keys server-side only; never embed them in browser or mobile clients.
  • Rotate keys regularly and immediately if compromise is suspected.

2.3 Request & response format

  • Only HTTPS is supported.
  • Requests must use Content-Type: application/json.
  • All responses are JSON objects.

3. Model card LexiMind Health Intelligence Model (HIM-2.0)

3.1 Intended use

  • Purpose: Support clinicians in stratifying risk for selected chronic conditions.
  • Users: Licensed clinicians and clinical support systems, via technical integrations.
  • Setting: Hospital and outpatient care; not intended for direct-to-consumer use.

3.2 Inputs & outputs

Aspect Details
Inputs Age, sex, blood pressure, glucose, cholesterol, risk factors (for example smoking, family history), and high-level symptoms where available.
Primary outputs Condition-specific risk scores (0–1), plus overall risk category (low, medium, high).
Secondary outputs Top contributing features used by the model (for interpretability and clinical discussion).

3.3 Behaviour & limitations

  • Optimised for adult populations (18+) with recent vitals and lab results.
  • Not validated for paediatrics, emergency triage, or rare diseases.
  • Predictions are probabilistic and must be treated as one signal in a wider clinical assessment.

4. Inference pipeline

Once a request passes authentication and basic validation, the following pipeline executes:

  1. Schema validation: Checks required fields, data types, and basic ranges.
  2. Normalisation: Harmonises units and encodings (for example mmHg, mg/dL, booleans).
  3. Feature engineering: Computes derived risk indicators and aggregates.
  4. Model inference (HIM-2.0): Produces per-condition risk scores.
  5. Risk categorisation: Maps numeric scores to discrete risk tiers.
  6. Response assembly: Packages scores, categories, and recommendation into JSON.

5. API endpoints

5.1 POST /predict

Main scoring endpoint. Accepts structured patient data and returns risk estimates for supported chronic conditions.

Method: POST
Path: /predict
Base URL: https://api.leximind.ai/v1

5.1.1 Request body example

{
  "patient_data": {
    "age": 55,
    "blood_pressure": 135,
    "cholesterol": 200,
    "blood_sugar": 110,
    "smoker": false,
    "family_history": ["diabetes"]
  }
}

5.1.2 Request fields

Field Type Required Description
patient_data.age integer required Age in years (18+ recommended).
patient_data.blood_pressure number required Systolic blood pressure in mmHg.
patient_data.cholesterol number required Total cholesterol in mg/dL.
patient_data.blood_sugar number required Fasting blood sugar level in mg/dL.
patient_data.smoker boolean optional Current smoking status.
patient_data.family_history array<string> optional List of relevant family conditions (for example "diabetes").

5.1.3 Response example

{
  "prediction": {
    "diabetes": 0.78,
    "heart_disease": 0.45,
    "kidney_disease": 0.12
  },
  "risk_level": "high",
  "recommendation": "Consult a clinician for diabetes-focused follow-up and monitoring.",
  "model_version": "HIM-2.0",
  "trace_id": "req_f91ec7d02a"
}

5.1.4 Response fields

Field Type Description
prediction object Per-condition risk scores between 0 and 1.
risk_level string Overall risk category (for example "low", "medium", "high").
recommendation string Plain-language guidance emphasising that predictions are a support signal, not a diagnosis.
model_version string Model identifier and version, for example "HIM-2.0".
trace_id string Unique identifier for this request, used for debugging and audits.

5.1.5 Python example

import requests

BASE_URL = "https://api.leximind.ai/v1"
API_KEY = "YOUR_API_KEY"

url = f"{BASE_URL}/predict"
headers = {"Authorization": f"Bearer {API_KEY}", "Content-Type": "application/json"}
payload = {
  "patient_data": {
    "age": 55,
    "blood_pressure": 135,
    "cholesterol": 200,
    "blood_sugar": 110,
    "smoker": False,
    "family_history": ["diabetes"]
  }
}

response = requests.post(url, headers=headers, json=payload, timeout=5)
response.raise_for_status()
print(response.json())

6. Error handling & retry strategy

Errors are communicated via standard HTTP status codes and a structured JSON body. Clients should branch behaviour based on error class: 

{
  "error": {
    "code": "4010_UNAUTHENTICATED",
    "message": "Missing or invalid API key.",
    "trace_id": "req_f91ec7d02a"
  }
}
HTTP code Error code Description Recommended action
400 4000_BAD_REQUEST Malformed JSON or invalid payload structure. Validate the request body and schema before retrying.
401 4010_UNAUTHENTICATED Missing or invalid API key. Check credentials and environment (staging vs production).
403 4030_FORBIDDEN Key exists but is not allowed to access this resource. Confirm organisational permissions and key scope.
429 4290_RATE_LIMIT Too many requests in a given time window. Implement exponential backoff and respect any Retry-After header.
500 5000_SERVER_ERROR Unexpected server-side failure. Retry with backoff; contact support with the trace_id if persistent.

7. Rate limits & performance expectations

7.1 Rate limits (illustrative)

  • Default: 100 requests per minute per API key.
  • Enterprise: Custom limits can be negotiated.

When limits are exceeded, the API returns HTTP 429 with a rate-limit error code. Clients should back off rather than continuously retry.

7.2 Latency targets

Endpoint Target P95 latency Notes
POST /predict < 800 ms Includes feature engineering and model inference.
GET /status < 150 ms Health check with no heavy computation.

8. Security, privacy & responsible use

8.1 Data protection

  • All traffic must use TLS (HTTPS); plaintext HTTP is not supported.
  • Only send features necessary for your clinical use case; avoid gratuitous data.
  • Where possible, avoid direct identifiers in patient_data and rely on internal IDs.

8.2 Responsible use guidance

  • Use outputs as decision support, not as standalone diagnoses.
  • Ensure clinicians retain final responsibility for clinical decisions.
  • Do not use predictions for insurance eligibility, pricing, or denial of care.

9. Integration checklist

Before going live, we recommend verifying the following:

  • API key securely stored and injected server-side.
  • Input validation implemented in your system (types, ranges, required fields).
  • Error handling covers 4xx vs 5xx vs 429 with appropriate retries and fallbacks.
  • Monitoring configured for latency, error rates, and usage per endpoint.
  • Clinical governance in place for reviewing and interpreting risk scores.
