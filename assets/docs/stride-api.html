Stride Platform API

A workflow-automation API: trigger runs over REST with bearer auth, and receive signed, at-least-once event webhooks as those runs progress built so retries never corrupt your data. The reference is organised around the one hard problem: consuming an at-least-once stream safely.

Overview

Stride runs automation workflows on your behalf. You start a run with a REST call; Stride executes it asynchronously and reports progress through events delivered to your webhook endpoints. The two channels have separate, purpose-built auth.

ChannelDirectionAuth
REST APIYou → StrideBearer token in the Authorization header.
WebhooksStride → YouSigned payloads you verify with a per-endpoint signing secret.

Run states

Run state machine
queued ──► running ──► succeeded
              │
              └─────► failed ──► (optional) retried ──► running …

Each transition emits an event: run.started, run.step.completed, run.succeeded, run.failed.

API authentication

Authenticate REST calls with a secret API token as a bearer credential. Tokens are scoped to a workspace and an environment (test or live).

Bearer auth
GET /v2/runs/run_8Kd2 HTTP/1.1
Host: api.stride.dev
Authorization: Bearer sk_live_3pQ9…vT
⚠ Treat live tokens as passwords

Never ship sk_live_ tokens to a browser or mobile client. For front-end use, mint short-lived scoped tokens server-side via POST /v2/tokens.

Idempotent writes

Any request that starts a run accepts an Idempotency-Key so a retried POST never starts the same run twice.

  • The first request with a key runs; its response is cached for 24 hours.
  • A repeat with the same key replays the cached response with Stride-Idempotent-Replay: true.
  • A repeat with the same key but a different body returns 409 idempotency_conflict.

Retrieve a run

GET/v2/runs/{run_id}

Polling is supported but secondary webhooks are the intended way to learn about completion. Use GET for reconciliation or after a missed event.

Webhooks

Stride POSTs events to your endpoint as they happen. Your endpoint must be public HTTPS and return any 2xx within 10 seconds. The response body is ignored; only the status matters.

Event shape

POST /your/webhook
POST /your/webhook HTTP/1.1
Stride-Signature: t=1733930535,v1=4f8c2a…e90b
Content-Type: application/json

{
  "id": "evt_2Qa7",
  "type": "run.succeeded",
  "created_at": "2025-12-11T14:02:15Z",
  "data": {
    "run_id": "run_8Kd2",
    "workflow_id": "wf_invoice_sync",
    "output": { "synced": 14, "skipped": 1 }
  }
}

Verifying the signature

The Stride-Signature header carries a timestamp (t) and one or more signatures (v1). Build the signed payload as t, a literal dot, and the raw request body. HMAC-SHA256 it with the endpoint's signing secret and compare in constant time.

Verification
signed_payload = t + "." + raw_body
expected = hex( HMAC_SHA256( signing_secret, signed_payload ) )
# accept if expected == v1 AND now - t <= 300 seconds
Worked example verifying an event
t (from header)1733930535
raw body{"id":"evt_2Qa7",…}
signed payload1733930535.{"id":"evt_2Qa7",…}
v1 (from header)4f8c2a…e90b
recompute & comparehex( HMAC_SHA256(secret, signed_payload) ) == v1 ✓
⚠ Verify the raw bytes

Verify against the raw bytes of the body, before any JSON parsing or re-encoding. Frameworks that hand you a re-serialized object will produce a different hash and fail verification.

Delivery & retries

Stride guarantees at-least-once delivery. A flaky network can produce duplicates, and events for one run can arrive out of order. Design consumers around both facts.

Deduplicate

Record processed event.id values and skip repeats.

Order by created_at

Order by created_at (and run state) rather than arrival time.

✅ Make handlers idempotent

Applying the same event twice must equal applying it once. This single property is what makes an at-least-once stream safe to consume.

Retry schedule

A non-2xx, a timeout, or a connection error schedules a retry on an exponential back-off with jitter. Each attempt carries the same event.id and a fresh signature.

Replaying missed events

Recover failed deliveries
GET  /v2/events?delivery=failed&since=2025-12-11T00:00:00Z
POST /v2/events/{event_id}/redeliver
ℹ Delivery status & retention

Endpoints surface a delivery status of delivered, pending, or failed. Failed events are retained for 30 days for replay.

Usage & credits

Runs consume credits from the workspace balance. Read the balance to surface usage in your own product or to gate work before you start it.

GET/v2/usage/credits
200 OK
HTTP/1.1 200 OK
Content-Type: application/json

{
  "balance": 8420,
  "included_monthly": 10000,
  "resets_at": "2026-01-01T00:00:00Z",
  "low_balance_threshold": 500
}

When the balance crosses low_balance_threshold, Stride emits a credits.low event so you can prompt an upgrade before runs start failing with 402 insufficient_credits.

✍️ Writer's note

The defining challenge of an event API isn't sending events it's helping the reader consume them safely. At-least-once delivery means duplicates and out-of-order arrival are normal, not exceptional, so I led the webhooks section with the three consumer rules (deduplicate, order by timestamp, idempotent handlers) before a single retry table. The signature-verification worked example spells out the one mistake everyone makes: hashing a re-serialized body instead of the raw bytes. Every product, endpoint, and identifier here is invented; no proprietary material is reproduced.
