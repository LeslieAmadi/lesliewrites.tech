1. Incident Summary

On 14 February 2025, NimbusTech experienced a major incident where the Payment Processing Service (NPPS) exhibited severe latency spikes followed by intermittent request failures. Approximately 31% of customer transactions failed or timed out during the incident window. The issue originated from a misconfigured database connection pool following a routine infrastructure scaling event.

The incident lasted 2 hours 36 minutes and was classified as Severity 1 (Critical) due to customer impact, financial transaction failure, and direct disruption to partner APIs.

3. Impact Assessment

  • Functional impact: NPPS API experienced 3–6 second latency spikes, followed by 500/504 errors.
  • Financial impact: Approx. 4,800 failed transactions; reconciliation required; estimated €27,000 in delayed settlements.
  • Compliance exposure: PSD2 response-time thresholds breached for EU partners.
  • Reputational impact: Multiple enterprise clients reported outages; support backlog increased by 42% during the incident.

5. Actions Taken

5.1 Immediate Actions

  • Increased API gateway timeouts from 3s → 6s to reduce cascading failures
  • Enabled degraded mode to route requests through cached transaction state

5.2 Corrective Actions

Database Engineering restored the cluster node count and rebalanced connections. SRE updated the connection pool configuration to prevent exhaustion during autoscaling.

5.3 Temporary Workarounds

A temporary Redis-based fallback was implemented to serve pre-authorised transactions while the database cluster stabilised.

7. Root Cause Analysis (RCA)

7.1 Root Cause

A misconfigured autoscaling policy reduced the database cluster size from 5 → 3 nodes during low overnight traffic. When traffic increased at peak hours, the reduced cluster capacity caused connection bottlenecks, draining the API’s connection pool and triggering cascading latencies.

7.2 Contributing Factors

  • Connection pool settings were tuned for a 5-node cluster only.
  • Autoscaler thresholds had not been validated against weekday peak load.
  • Lack of alerting on unexpected infrastructure downscaling.

7.3 5 Whys Analysis

  1. Why? → Because the database cluster unexpectedly scaled down.
  2. Why? → Because autoscaling thresholds were too aggressive.
  3. Why? → Because they were configured for weekend patterns, not weekdays.
  4. Why? → Because the autoscaling policy was updated without SRE review.
  5. Why? → Because change-management rules did not include cross-team review for scaling policies.

7.4 Evidence & Logs

Included in internal Jira ticket: DB-1228 (not publicly shared).

8. Long-Term Preventive Measures

  • Implement SRE-approved autoscaling policy with min-node lock.
  • Deploy real-time alerts for infrastructure downscaling events.
  • Introduce automated connection pool tuning based on cluster size.
  • Mandatory cross-team review for infra configuration changes.
  • Monthly resilience testing for all payment-related databases.

9. Post-Incident Validation

9.1 Monitoring Results

Latency averages returned to 200–300ms; error rate <0.04%.

9.2 QA Validation

Regression tests passed for all payment flows.

9.3 Customer Confirmation

All enterprise banking partners confirmed stable recovery and successful transaction retries.
