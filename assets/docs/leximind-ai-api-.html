Technical API & System Documentation · Author: Leslie Amadi

The LexiMind Predictive Diagnosis API exposes a clinical decision-support model that estimates the likelihood of chronic conditions based on structured patient data. It is designed to integrate into electronic health record (EHR) systems, hospital information systems, and analytics platforms as a recommendation engine, not as an autonomous diagnostic tool.

Service Name: predictive-diagnosis-api
Owner: Clinical ML Platform Team
Primary Users: EHR integrators, data engineers, clinical analytics teams
Status: Production, Model v2.0
Scope. This document provides a senior-level overview of the Predictive Diagnosis API, including architecture, model characteristics, API contracts, performance expectations, error handling, security and compliance considerations, and guidance for responsible clinical integration.

1. Environments & Authentication

1.1 Base URLs

The API is deployed in separate environments:

  • Staging: https://staging.api.leximind.ai/v1
  • Production: https://api.leximind.ai/v1

All examples in this document use the production base URL. For testing, replace the host with the staging endpoint.

1.2 Authentication

LexiMind uses API key–based authentication. Each request must include the Authorization header: 

Authorization: Bearer YOUR_API_KEY

API keys are organisation-scoped and environment-specific. Keys must be stored server-side and not embedded in client-side code or public repositories. Rotate keys regularly and immediately on suspected compromise.

1.3 Request & Response Format

  • All requests must use HTTPS.
  • Content type: application/json.
  • Responses are returned as JSON objects.
  • All timestamps use ISO 8601, UTC.

2. System Architecture

The Predictive Diagnosis service is part of the broader LexiMind platform and follows a modular, API-first architecture. It is stateless at the HTTP layer and relies on dedicated model-serving infrastructure for inference.

2.1 Component Overview

  • API Gateway – Validates requests, enforces authentication, rate limits and basic schema checks.
  • Diagnosis Service – Accepts validated payloads, orchestrates feature extraction and model inference.
  • Model Serving Layer – Hosts predictive models (e.g. gradient boosting / deep nets) with versioned deployments.
  • Feature Store – Provides standardised clinical features and derived variables (e.g. BMI categories, comorbidity flags).
  • Logging & Telemetry – Captures structured logs, metrics and traces for monitoring and auditability.
  • Feedback Ingestion – Receives optional ground-truth labels and clinician feedback for offline evaluation.

3. Model Card Predictive Diagnosis v2.0

3.1 Intended Use

  • Primary purpose: Support clinicians in identifying patients at higher risk of selected chronic conditions.
  • Intended users: Licensed clinicians and clinical decision-support systems.
  • Intended setting: Hospital and outpatient care, not direct-to-consumer.

3.2 Inputs & Outputs

  • Inputs: age, sex, vitals, lab values, risk factors, symptoms
  • Outputs: per-condition risk scores (0–1), discrete risk level, feature attributions

3.3 Training Data (High-Level)

  • De-identified EHR data from multiple partner hospitals.
  • Adult patients (18+), diverse demographics and comorbidity profiles.
  • Label definitions based on ICD codes and clinician-validated cohorts.

Not trained on paediatric populations or rare disease cohorts.

3.4 Evaluation Metrics (Illustrative)

Condition ROC AUC PR AUC Sensitivity @ fixed specificity
Type 2 Diabetes 0.89 0.64 0.81 @ 0.80
Chronic Kidney Disease 0.86 0.57 0.78 @ 0.80
Coronary Heart Disease 0.84 0.52 0.76 @ 0.80

3.5 Limitations & Exclusions

  • Not validated for paediatrics, obstetrics, psychiatry, or emergency triage.
  • Performance may degrade on populations that differ significantly from the training cohort.
  • Predictions should not be used for insurance pricing, eligibility, or administrative denial of care.

4. Data & Inference Pipeline

The inference pipeline converts raw JSON payloads into structured clinical features consumed by the model.

  1. Input validation – Required fields checked, types validated, ranges sanity-checked.
  2. Normalisation – Units harmonised (e.g. mmHg, mg/dL), categorical values standardised.
  3. Feature engineering – Derived risk factors computed (e.g. BMI categories, multi-morbidity indices).
  4. Model inference – Condition-specific models invoked, producing risk scores per condition.
  5. Risk stratification – Scores mapped to discrete levels (low, medium, high).
  6. Explanations – Top contributing features returned for interpretability.

   -> normalise units & categories
   -> feature engineering (derived signals)
   -> model inference (per condition)
   -> risk stratification (low / medium / high)
   -> explanations + response assembly

5. API Design & Endpoints

The Predictive Diagnosis API follows a JSON-over-HTTP design, with clear, versioned endpoints.

5.1 POST /predict-diagnosis

Submits structured patient data and returns a risk score and risk level for one or more chronic disease categories.

Method: POST
Path: /predict-diagnosis

Request Body Example

{
  "patient_id": "A-49302",
  "age": 56,
  "sex": "female",
  "bmi": 29.4,
  "blood_pressure": 138,
  "glucose": 108,
  "cholesterol": 210,
  "smoker": false,
  "family_history": ["diabetes", "hypertension"],
  "symptoms": ["fatigue", "shortness_of_breath"]
}

Request Fields

Field Type Required Description
patient_id string optional Local or anonymised identifier for traceability. Avoid direct identifiers (e.g. MRNs) where possible.
age integer required Age in years (18+ recommended).
sex string required "male", "female", or "other".
bmi number optional Body Mass Index.
blood_pressure number required Systolic blood pressure (mmHg).
glucose number required Fasting glucose level (mg/dL).
cholesterol number optional Total serum cholesterol (mg/dL).
smoker boolean optional Current smoking status.
family_history array<string> optional Relevant conditions in family history (e.g. "diabetes", "hypertension").
symptoms array<string> optional List of reported symptoms (free-text category codes).

Response Example

{
  "risk_score": 0.81,
  "risk_level": "high",
  "conditions": {
    "diabetes_type_2": 0.78,
    "chronic_kidney_disease": 0.42,
    "coronary_heart_disease": 0.35
  },
  "model_version": "2.0",
  "explanations": {
    "top_positive": ["blood_pressure", "age", "family_history.diabetes"],
    "top_negative": ["glucose"]
  },
  "trace_id": "req_9f3a92b7c1"
}

Response Fields

Field Type Description
risk_score number Overall risk estimate between 0 and 1.
risk_level string One of "low", "medium", "high".
conditions object Per-condition risk scores keyed by condition identifier.
model_version string Deployed model version used for this prediction.
explanations object Key features influencing the prediction (e.g. top positive/negative contributors).
trace_id string Identifier for auditing, debugging and feedback correlation.

Python Example

import requests

BASE_URL = "https://api.leximind.ai/v1"
API_KEY = "YOUR_API_KEY"

payload = {
  "patient_id": "A-49302",
  "age": 56,
  "sex": "female",
  "bmi": 29.4,
  "blood_pressure": 138,
  "glucose": 108,
  "cholesterol": 210,
  "smoker": False,
  "family_history": ["diabetes", "hypertension"],
  "symptoms": ["fatigue", "shortness_of_breath"]
}

headers = {
  "Authorization": f"Bearer {API_KEY}",
  "Content-Type": "application/json"
}

response = requests.post(f"{BASE_URL}/predict-diagnosis", headers=headers, json=payload)
print(response.status_code, response.json())

6. Performance & Monitoring

Performance is monitored along latency, reliability and prediction quality dimensions. Metrics are exported to central monitoring dashboards.

6.1 Latency Targets (Illustrative)

Metric P95 Target Notes
End-to-end latency: /predict-diagnosis < 800 ms Includes feature engineering and model inference.
End-to-end latency: /status < 150 ms Simple readiness probe, no model calls.
5xx error rate < 0.5% Rolling 7-day window.

6.2 Observability

  • Logging: Structured JSON logs with trace IDs; sensitive values minimised or redacted.
  • Metrics: Request counts, latency histograms, error codes, model-version usage.
  • Tracing: Distributed traces for request path: gateway → diagnosis service → model serving.

7. Error Handling & Retry Strategy

LexiMind returns standard HTTP status codes together with a structured error payload: 

{
  "error": {
    "code": "4001_INVALID_SCHEMA",
    "message": "Request body does not match expected schema.",
    "trace_id": "req_9f3a92b7c1"
  }
}
HTTP Code Error Code Description Recommended Client Action
400 4001_INVALID_SCHEMA Request body does not match expected schema. Validate JSON structure and required fields before retrying.
401 4010_UNAUTHENTICATED Missing or invalid API key. Check token, environment and authentication configuration.
403 4030_FORBIDDEN API key does not have permission for this resource. Confirm key scope and organisation access settings.
429 4290_RATE_LIMIT Too many requests in a given window. Throttle requests; implement exponential backoff using Retry-After where present.
500 5001_SERVER_ERROR Unexpected internal error. Retry with backoff; contact support if persistent and include trace_id.

8. Security, Privacy & Compliance

8.1 Data Protection

  • All traffic is encrypted in transit via TLS 1.2+.
  • Only the minimum necessary clinical features should be sent for a given use case.
  • Direct identifiers (names, addresses) should be excluded where possible.

8.2 Compliance

  • Service is designed to be compatible with HIPAA/GDPR-aligned environments.
  • Data controllers remain responsible for legal basis and consent management.
  • Log retention windows are configurable; default retention is time-bound and access-controlled.

8.3 Access Control

  • API keys are scoped to organisations; misuse can be traced and revoked.
  • Administrative access to logs and telemetry is restricted to authorised operations and ML teams.

9. Risk Management & Responsible Use

9.1 Human-in-the-Loop

  • Predictions are decision support only; clinicians must retain ultimate responsibility.
  • Model outputs should be reviewed in context of full clinical history and additional diagnostics.

9.2 Model Drift & Monitoring

  • Input distributions and calibration metrics are periodically monitored.
  • Significant drift triggers re-evaluation and potential retraining.

9.3 Fairness & Bias

  • Model performance is evaluated on key demographic subgroups where data permits.
  • Clients should avoid using risk scores in ways that could reinforce structural inequities.

9.4 Prohibited Uses

  • Insurance coverage, pricing, or eligibility decisions.
  • Autonomous triage without human oversight.

10. Versioning & Roadmap

10.1 Versioning

LexiMind follows a semantic-versioning inspired model:

  • Major (2.x → 3.x): may introduce breaking changes or new condition sets.
  • Minor (2.0 → 2.1): backwards-compatible improvements (e.g. calibration, additional explanations).
  • Patch (2.0.0 → 2.0.1): bug fixes and internal improvements.

API responses include a model_version field so clients can track which model served a request.

10.2 Roadmap (Illustrative)

  • Support for additional chronic conditions with sufficient training data.
  • Richer explainability (e.g. feature contribution magnitudes, confidence bands).
  • Configurable thresholds per organisation or clinical pathway.

11. Integration Checklist & Support

11.1 Integration Checklist

  • Obtain and securely store API key (per environment).
  • Implement input validation and unit normalisation upstream where possible.
  • Propagate trace_id in logs for debugging.
  • Implement safe fallbacks if the API is temporarily unavailable (e.g. continue workflow without risk score).
  • Ensure local governance and clinical review for deployment decisions.
