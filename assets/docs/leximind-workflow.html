1. Purpose

This workflow defines the standardised process for managing and transitioning tickets across Booming Games’ integration, QA, and support lifecycle. It ensures:

  • Consistent and auditable transitions between teams.
  • Clear entry and exit criteria for every status.
  • SLA-aligned movement to support customer integrations.
  • Cross-department alignment between HOCS, CS, QA, Legal, and Product.

2. Scope

This workflow applies to:

  • HOCS – workflow initiation, governance, and customer coordination.
  • Customer Success (CS) – integration support and customer communication.
  • QA – functional validation, regression, defect reporting.
  • Legal – contract compliance (when needed).
  • Product / Dev – escalations, blockers, engineering fixes.

3. Workflow Overview

Core workflow path used for all partner integrations:

To Do 
  → Client Setup 
  → Pending Customer Feedback 
  → Customer Integration 
  → QA Testing 
  → Ready to Go Live 
  → Live

4. Status Definitions

Each status includes meaning, entry criteria, and required exit criteria.

Status Meaning Entry Criteria Exit Criteria
To Do Initial creation of task or integration request. Ticket created with baseline requirements. HOCS confirms customer and product involved.
Client Setup Credentials, channel setup, environment preparation. Customer contact + product information confirmed. Credentials generated, documented, and delivered securely.
Pending Customer Feedback Customer must provide input or approval. CS sends request and logs communication. Customer responds with next steps or approval.
Customer Integration Customer is actively implementing the integration. Test credentials in place. Customer confirms readiness for QA.
QA Testing Functional testing + integration validation. Customer indicates integration is complete. All tests passed, defects resolved, test results logged.
Ready to Go Live Integration approved for production. QA sign-off is completed and documented. Production configuration staged and validated.
Live Integration is active in production. Go-live executed. Stability monitoring completed (24–48 hours).

5. Transition Rules

5.1 Allowed Transitions

From To Actor Conditions
To Do Client Setup HOCS Kick-off call completed.
Client Setup Pending Customer Feedback CS / HOCS Credentials delivered, awaiting customer action.
Customer Integration QA Testing CS Customer confirms readiness.
QA Testing Ready to Go Live QA All tests passed.
Ready to Go Live Live HOCS / CS Production configuration approved.

5.2 Forbidden Transitions

From To Reason
Live QA Testing Post-release tests must be logged as new tickets.
Ready to Go Live Customer Integration Backwards movement violates audit trail.

6. SLA Expectations

Step SLA Owner
Customer Integration 7 days (customer-dependent) Customer
QA Testing 2–3 business days QA Team
Pending Customer Feedback Reviewed every 48–72 hours HOCS / CS

7. Role Responsibilities

HOCS

  • Initiates workflow and ensures compliance.
  • Runs kick-off and onboarding calls.
  • Determines transition readiness.

Customer Success (CS)

  • Supports customer through integration.
  • Moves tickets into QA readiness.
  • Documents customer communication.

QA Team

  • Executes testing and logs results.
  • Approves or blocks go-live transitions.

Customer

  • Completes integration work.
  • Provides timely feedback.

8. RACI Matrix

Action R A C I
Move ticket to Client Setup HOCS HOCS CS QA
Begin QA Testing QA QA Lead CS HOCS
Approve Go Live HOCS HOCS CS, QA Legal
