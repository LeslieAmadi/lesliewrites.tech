1. Overview

The LexiMind Predictive Diagnosis API exposes a machine learning model that estimates chronic disease risk based on structured patient data. It is designed for integration into clinical dashboards, decision-support tools, and analytics pipelines.

This reference focuses on the core predictive endpoint POST /v1/predict-diagnosis, detailing its request and response schema, validation behaviour, error model, rate limits, and operational considerations.

Clinical Responsibility Note: LexiMind predictions are designed for clinical decision support and must not be used as standalone medical diagnoses. A licensed healthcare professional remains responsible for final medical decisions.

2. Environments & Base URLs

The API is deployed in two environments:

Environment Base URL Usage
Staging https://staging.api.leximind.ai/v1 Integration testing, non-production workloads
Production https://api.leximind.ai/v1 Live clinical and analytics workloads

All examples in this document assume the production base URL. To use the staging environment, replace the host accordingly.

3. Authentication

LexiMind uses Bearer token authentication. Every request must include a valid API key: 

Authorization: Bearer YOUR_API_KEY
  • API keys are organisation-scoped and environment-specific.
  • Do not expose keys in public front-end code or client-side applications.
  • Rotate keys regularly according to your security policy.

6. Endpoint: POST /v1/predict-diagnosis

6.1 Description

Accepts structured patient data and returns an overall risk score, risk level and per-condition probabilities for selected chronic illnesses. The endpoint is synchronous and optimised for real-time usage in clinical workflows.

6.2 HTTP Details

Property Value
MethodPOST
Path/v1/predict-diagnosis
Content-Typeapplication/json
Acceptapplication/json
Timeout (recommended)5–8 seconds

6.3 Request Body Example

{
  "patient_id": "A-49302",
  "age": 56,
  "sex": "female",
  "bmi": 29.4,
  "blood_pressure": 138,
  "glucose": 108,
  "cholesterol": 210,
  "smoker": false,
  "family_history": ["diabetes", "hypertension"],
  "symptoms": ["fatigue", "shortness_of_breath"]
}

6.4 Request Fields

Field Type Required Description
patient_id string No Local or anonymised identifier used for traceability.
age integer Yes Age in years. Range: 0–120.
sex string Yes "male", "female", or "other".
bmi number No Body Mass Index (kg/m²). Optional but improves accuracy.
blood_pressure number Yes Systolic blood pressure (mmHg).
glucose number Yes Fasting glucose level (mg/dL or mmol/L normalised server-side).
cholesterol number No Total cholesterol (mg/dL).
smoker boolean No Indicates whether the patient is a current smoker.
family_history array<string> No List of relevant conditions (e.g. "diabetes", "hypertension").
symptoms array<string> No Reported symptoms, free text or controlled vocabulary.

6.5 Validation Rules

Field Rule Failure Behaviour
age 0 ≤ age ≤ 120 400 with INVALID_RANGE
sex One of ["male","female","other"] 400 with INVALID_ENUM
blood_pressure Greater than 0 400 with INVALID_VALUE
glucose Greater than 0 400 with INVALID_VALUE

7. Response Schema

7.1 Successful Response (200 OK)

{
  "risk_score": 0.81,
  "risk_level": "high",
  "conditions": {
    "diabetes_type_2": 0.78,
    "chronic_kidney_disease": 0.42,
    "coronary_heart_disease": 0.35
  },
  "model_version": "2.0",
  "explanations": {
    "top_positive": ["blood_pressure", "age", "family_history.diabetes"],
    "top_negative": ["glucose"]
  },
  "trace_id": "req_9f3a92b7c1"
}

7.2 Response Fields

Field Type Description
risk_score number Overall risk estimate between 0 and 1.
risk_level string Categorical label, e.g. "low", "medium", "high".
conditions object Per-condition probabilities keyed by condition name.
model_version string Identifier of the model used for this prediction.
explanations object Top positive and negative contributing features (for interpretability).
trace_id string Unique identifier for tracing and audit logs.

8. Error Handling

All errors are returned with a standard JSON payload and appropriate HTTP status.

8.1 Error Response Format

{
  "error": {
    "code": "4001_INVALID_SCHEMA",
    "message": "Request body does not match expected schema.",
    "details": {
      "field": "age",
      "issue": "required"
    },
    "trace_id": "req_9f3a92b7c1"
  }
}

8.2 Error Catalogue

HTTP Status Error Code Description Typical Resolution
400 4001_INVALID_SCHEMA Request body is malformed or missing required sections. Validate JSON structure against the documented schema.
400 4002_MISSING_FIELDS Required fields such as age or blood_pressure are absent. Add the missing fields and retry.
401 4010_UNAUTHENTICATED Missing or invalid bearer token. Check API key configuration and token validity.
403 4030_FORBIDDEN API key does not have access to this endpoint or environment. Confirm key scope and environment mapping.
429 4290_RATE_LIMIT Too many requests within the configured time window. Throttle calls or implement exponential backoff.
500 5001_MODEL_UNAVAILABLE Model backend temporarily unavailable. Retry with backoff; contact support if persistent.

9. Rate Limits

Default rate limits for the LexiMind Predictive Diagnosis API are:

Plan Per-Minute Limit Daily Limit
Standard 100 requests / minute 10,000 requests / day
Enterprise Custom, contract-defined Custom, contract-defined

When a client exceeds rate limits, a 429 response is returned with 4290_RATE_LIMIT. Implement retry logic with exponential backoff to avoid repeated failures.

10. Security & Compliance

  • All traffic is encrypted via HTTPS (TLS 1.2+ only).
  • API keys must be stored server-side, never in unencrypted client code.
  • Access logs and trace IDs can be used for audit trails.
  • PHI/PII handling must comply with local regulatory requirements.
Note: LexiMind does not persist raw patient payloads by default. Optional logging for debugging must be explicitly enabled and configured in line with your organisation’s data retention policies.

11. Usage Patterns & Best Practices

  • Validate inputs on your side before calling the API to reduce 4xx errors.
  • Log trace_id values for troubleshooting with LexiMind support.
  • Combine API outputs with existing clinical risk scores, not as a replacement.
  • Cache non-patient-specific results (e.g., metadata) where appropriate.

12. Quickstart Examples

12.1 cURL Example

curl -X POST "https://api.leximind.ai/v1/predict-diagnosis" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "patient_id": "A-49302",
    "age": 56,
    "sex": "female",
    "bmi": 29.4,
    "blood_pressure": 138,
    "glucose": 108,
    "cholesterol": 210,
    "smoker": false
  }'

12.2 Python Example

import requests

BASE_URL = "https://api.leximind.ai/v1"
API_KEY = "YOUR_API_KEY"

payload = {
  "patient_id": "A-49302",
  "age": 56,
  "sex": "female",
  "bmi": 29.4,
  "blood_pressure": 138,
  "glucose": 108,
  "cholesterol": 210,
  "smoker": False
}

headers = {
  "Authorization": f"Bearer {API_KEY}",
  "Content-Type": "application/json"
}

response = requests.post(f"{BASE_URL}/predict-diagnosis",
                         headers=headers,
                         json=payload)

print(response.status_code)
print(response.json())
