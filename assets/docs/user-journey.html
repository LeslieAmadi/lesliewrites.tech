User Journey Mapping

A journey map turns “we think the experience is fine” into a shared, evidence-based picture of what a real person actually does, thinks, and feels and exactly where it breaks. I built the documentation system for creating maps that drive decisions: personas, stages, actions, the emotion curve, pain points, and the disciplined step from insight to prioritised opportunity.

The business problem

A product team is convinced onboarding is “pretty smooth.” Support sees a different reality: a wave of tickets every time a new user hits the same invisible wall on day two. Sales hears “it was confusing at first.” Each team holds a fragment of the truth, nobody holds the whole, and the roadmap gets argued from opinion because there's no shared picture of the actual experience.

What a journey map fixes

The problem isn't a lack of data it's that the data is scattered across teams and nobody has assembled it into one view of what a real person goes through, end to end. A journey map is that single view. The documentation I built makes the map a rigorous, repeatable artifact grounded in evidence, not a sticky-note exercise that gets photographed and forgotten.

The anatomy of a journey map

A journey map is a grid: stages run left to right across the top, and a set of lanes runs down the side. Every map the documentation produces uses the same lanes, so any reader can read any map.

LaneAnswersSourced from
StagesWhat phases does the journey move through?Research, analytics funnels
ActionsWhat does the person actually do at each stage?Observation, session data
ThoughtsWhat are they trying to achieve or work out?Interviews, surveys
EmotionHow do they feel and how does that change?Interviews, support tone
TouchpointsWhat do they interact with (app, email, support)?Product, CRM, support logs
Pain pointsWhere does it break or frustrate?Tickets, drop-off, reviews

The emotion curve

The emotion curve is the heartbeat of the map. Plotting how a person feels across the stages turns a flat table into a story, and the dips point straight at where intervention matters most.

A journey map with the emotion curve plotted across stages. The deepest dip here, a confusing empty first screen is the highest-leverage place to act, because it's where a real person is most at risk of leaving.

Read the dips, not the average

A journey that's “mostly fine” with one sharp low point is not a fine journey the low point is where people abandon. Like slice analysis in model evaluation, the average hides the failure; the map's value is that it makes the worst moment impossible to ignore.

🔧

Evidence, not assumption

The fastest way to discredit a journey map is to fill it with guesses. The single rule that separates a useful map from a workshop artifact is that every cell cites a source.

The two-colour discipline

In the template, evidence-backed entries and assumptions are visually distinct. Assumptions are allowed you rarely have data for everything but they're marked as hypotheses to validate, never presented as fact. This keeps the map honest and turns its blank spots into a research backlog rather than hidden risk.

💡

From insight to opportunity

This is the step most journey maps skip, and the reason so many end up as wall art. A pain point is only useful if it becomes an ownable, prioritised opportunity. The documentation makes that translation a required final stage.

Pain pointOpportunity (“How might we…”)Owner
Stuck on an empty first screenHow might we show a meaningful first action within 30 seconds of first login?Product
Unsure if sign-up workedHow might we confirm success immediately and set expectations for next steps?Design
Repeat “how do I…” ticketsHow might we answer the top three questions in-product, at the moment they arise?Content / Support
Prioritise by depth × reach

Not every pain point is worth fixing now. The documentation prioritises opportunities by how deep the emotional dip is and how many people hit it the deep, common dips first. This gives product a defensible, evidence-based order rather than a list of everything that's wrong.

🤝

Running the mapping workshop

A map built alone is one person's opinion. The facilitation guide makes it a cross-team artifact built with the people who hold the evidence.

StepWhat happens
1 · FrameAgree the one persona and one scenario this map covers. Resist mapping “everyone.”
2 · StagesLay out the journey phases together, end to end.
3 · PopulateFill actions, thoughts, emotion, touchpoints each with its source.
4 · Find the dipsPlot the emotion curve and mark pain points where it drops.
5 · OpportunitiesConvert each pain point into a “how might we” with an owner.
6 · PrioritiseRank by depth × reach; hand the top items to the roadmap.

Key learnings

One persona, one scenario.

Maps that try to cover everyone cover no one. The sharpest insights always came from the narrowest, most specific map.

The map is a means, not the deliverable.

The real deliverable is the prioritised opportunity list. A beautiful map with no opportunities changed nothing.

Cite every cell.

The two-colour evidence-vs-assumption discipline was what earned the map credibility with skeptical stakeholders.

📝 Writer's note

Journey maps are easy to make pretty and hard to make useful. I built this documentation to fight the most common failure mode the map that gets celebrated in a workshop and then never influences a single decision.

The two moves that mattered most were forcing every cell to cite evidence, and making the insight-to-opportunity translation a required stage rather than an optional epilogue. Those turned the map from a picture of the present into a plan for the future, which is the only version worth a team's time.
