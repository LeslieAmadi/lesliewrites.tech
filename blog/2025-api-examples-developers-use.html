API Examples Developers Actually Use

Most API examples look complete on the page. They have the right syntax, the right structure, the right shape. Then a developer copies one, runs it, and nothing works. A missing header. A placeholder nobody replaced. A response that bears no resemblance to what the endpoint actually returns. Any one of these transforms a helpful example into a frustration and a frustrated developer trusts your documentation a little less from that moment on.

There is a moment I have seen play out many times across developer onboarding sessions, internal documentation reviews, and support queues: an engineer reads an API example, tries to run it, and something breaks. They spend ten, fifteen, sometimes thirty minutes investigating. Eventually they discover the issue was not in their code or their environment it was in the example itself. A value that was never realistic. A header that was quietly omitted. A response that was invented rather than captured from the actual API.

They fix the problem and move on. But something has shifted. The documentation has become a source of suspicion rather than confidence. Every example that follows gets scrutinised a little more carefully. Every response gets cross-checked. The cognitive overhead of working with the API quietly doubles.

This is the real cost of poor API examples not the fifteen minutes lost debugging a single call, but the tax placed on every interaction with the documentation that follows. Writing examples that actually work is one of the highest-leverage investments a technical writer can make in developer experience. Here is what that looks like in practice.

The structural problem: examples written from memory

Most bad API examples share the same origin story. A developer or technical writer constructs an example from their knowledge of the API rather than from a live call against a real endpoint. They know the basic structure, they know which fields are required, they know roughly what a response looks like. They write something that looks correct.

The problem is that "looks correct" and "is correct" diverge in ways that only become visible when someone actually runs the code. Required headers get forgotten because they are so automatic to the person writing the example that they don't think to include them. Response fields appear in the wrong order, or include fields that don't exist, or omit fields that always appear. Error codes are invented rather than documented from the actual error responses the API returns.

The discipline that prevents this is simple and non-negotiable: every example in your documentation should be generated by running the actual call against a real test environment, capturing the actual response, and using those as the basis for the documented example. Not a reconstruction. The real thing.

The only acceptable test

Before any API example is published, a person who has never worked on that endpoint should be able to copy the example exactly as written, run it against the test environment, and receive a response that matches the documented response. If they cannot, the example is not ready.

Placeholder values: the single most common failure

Placeholder values YOUR_API_KEY, <ACCESS_TOKEN>, INSERT_CUSTOMER_ID_HERE are the most common source of developer frustration in API documentation. They are used with good intentions: the writer knows they cannot publish real credentials, so they mark the spot where the developer needs to substitute their own.

The problem is that placeholders communicate nothing about what should go there. What format is the API key? How long is it? What characters does it contain? Where do I find it? A developer who has never used this API before cannot answer any of these questions from the placeholder alone.

The solution is realistic test values values that are obviously fictional but structurally authentic. A good test API key looks like a real API key from the same system. It follows the same prefix convention, the same length, the same character set. It sets expectations. It shows the developer what to look for when they retrieve their own credentials.

❌ Placeholder communicates nothing

Authorization: Bearer YOUR_API_KEY

Developer copies this. Gets a 401. Spends fifteen minutes wondering if their key is wrong before realising the placeholder is the problem. They are now frustrated and slightly less trusting of the documentation.

✅ Realistic test value sets clear expectations

Authorization: Bearer sk_test_51Lx9C0Xexample123

Developer immediately understands the key format: the sk_test_ prefix, the approximate length, the character set. They know their own key will look similar. They replace this value confidently.

The test values you use in examples should be obviously fictional a developer should never mistake them for real credentials but they should be structurally indistinguishable from real values at a glance. The format is the information.

A complete example: request and response together

An API example that shows only the request is half an example. Developers need to see what success looks like before they can evaluate whether their own implementation is working correctly. Without a documented expected response, a developer who receives any response has no way to know if it is the right one.

Here is what a complete payment creation example looks like when it is written with the developer's experience in mind. Every field is present. The response matches what the endpoint actually returns. The values are realistic enough to set accurate expectations about format and type. 

curl -X POST "https://api.example.com/v1/payments" \
     -H "Authorization: Bearer sk_test_51Lx9C0Xexample123" \
     -H "Content-Type: application/json" \
     -H "Idempotency-Key: idem_key_20250312_001" \
     -d '{
       "amount": 5000,
       "currency": "EUR",
       "customer_id": "cus_00123",
       "description": "Invoice #1042",
       "metadata": {
         "order_id": "ord_789"
       }
     }'

Expected response HTTP 201 Created:

{
  "id": "pay_98765",
  "object": "payment",
  "status": "succeeded",
  "amount": 5000,
  "currency": "EUR",
  "customer_id": "cus_00123",
  "description": "Invoice #1042",
  "metadata": {
    "order_id": "ord_789"
  },
  "idempotency_key": "idem_key_20250312_001",
  "created_at": "2025-03-12T10:15:30Z",
  "updated_at": "2025-03-12T10:15:30Z"
}

Notice that the response includes every field from the request, plus the fields the API adds: the payment ID, the object type, the status, timestamps, and the idempotency key echo. A developer who sees this response in the documentation can write their parsing logic, their success handling, and their logging before they have made a single real API call. That is documentation doing its job.

Error responses deserve the same care

Most API documentation shows the happy path and stops there. This is a significant omission. Developers spend a substantial proportion of their integration time handling errors. Authentication failures, validation errors, rate limits, network timeouts these are not edge cases. They are the expected experience of building against a real API. If your documentation does not show what errors look like, you are leaving developers to guess.

Guessing at error shapes leads to fragile error handling. A developer who does not know your error response format writes code that breaks the first time they hit a 422. A developer who has seen your error format documented can write robust error handling before they encounter a single real failure. 

# Authentication failure — 401 Unauthorized
{
  "error": {
    "code": "authentication_failed",
    "message": "The API key provided is invalid, expired, or has been revoked.",
    "doc_url": "https://docs.example.com/errors/authentication_failed",
    "status": 401
  }
}

# Validation error — 422 Unprocessable Entity
{
  "error": {
    "code": "validation_error",
    "message": "Request validation failed.",
    "param": "amount",
    "detail": "The 'amount' field is required and must be a positive integer representing the amount in the smallest currency unit.",
    "doc_url": "https://docs.example.com/errors/validation_error",
    "status": 422
  }
}

# Rate limit exceeded — 429 Too Many Requests
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "You have exceeded the rate limit of 100 requests per minute.",
    "retry_after": 23,
    "doc_url": "https://docs.example.com/errors/rate_limit_exceeded",
    "status": 429
  }
}

Three error cases. Each one tells the developer exactly what went wrong, points to extended documentation, and in the rate limit case tells them exactly how long to wait before retrying. This is error documentation that respects the developer's time and treats them as capable of handling failures gracefully when given the right information.

The idempotency conversation no one is having

One of the most underserved topics in API documentation is idempotency the property that allows a request to be safely retried if the response is uncertain. When a developer sends a payment request and the network drops before they receive a response, they face a critical question: did the payment go through? If they retry without an idempotency key, they may create a duplicate charge. If they do not retry, they may miss a successful payment.

Idempotency keys solve this, but only if developers know about them and understand how to use them. If your API supports idempotency, document it prominently in the quickstart, in the payment creation example, and in its own dedicated guide. The cost of a developer not understanding idempotency is measured in duplicate charges, angry customers, and support escalations. The cost of documenting it well is a few hours of writing time.

Code samples in multiple languages

curl is the universal language of API documentation it is readable, it is language-agnostic, and it translates directly to any HTTP client. But many developers integrating your API will be working in Python, JavaScript, Go, or Ruby. Providing examples in the languages your users actually work in dramatically reduces the cognitive overhead of integration.

A Python developer looking at a curl example is doing mental translation work throughout. They know the curl command, they understand the API, but they are still one layer of abstraction away from the code they need to write. A Python example removes that layer. The same payment creation call, in the language they are working in, gets them from documentation to running code in significantly less time.

You do not need examples in every language. Identify the two or three languages your users most commonly use this is almost always visible in your support queries and community discussions and provide working examples in those. Maintained examples in three languages are worth more than unmaintained examples in ten.

The test that cannot be skipped

Every API example in your documentation should pass a simple test: a developer who has never used this API before should be able to copy the example exactly as written, substitute their own test credentials, and receive a successful response without needing to look anything else up.

Running this test yourself, as the technical writer, is not optional. It is the work. The five minutes it takes to copy your own example, run it against a test environment, and verify the response is the investment that determines whether the documentation is trustworthy. Every time you skip that step, you are making a bet that no one will notice the difference. In my experience, someone always does.
